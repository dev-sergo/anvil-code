# Design — v1.33 reranker

> **Status:** DRAFT — pre-impl. **Date:** 2026-05-07.
> **Order:** идёт после v1.32-c.1 (Phase 3 closure). Phase 4 первая итерация.
> **Links:** [v1.32-c.1 design](v1.32-c.1-no-progress-nudge.md), [llama-api-reference](../llama-api-reference.md).

## TL;DR

Текущий retrieval делает один cosine-similarity hop: `HNSW.search(queryVec, k=5)`. На больших репо (rag-system-target, 91 файл) dense-only даёт noise — похожие сигнатуры и vocab gap сдвигают релевантные символы из top-5. Добавляем BGE-reranker-v2-m3 как второй pass: HNSW возвращает top-30 кандидатов, reranker (cross-encoder, видит query+doc вместе) переранжирует, берём top-5. Инфраструктура готова: `POST /v1/rerank` на llama-swap работает, модель загружена (~418 MB, VRAM не конкурирует с qwen-coder-long).

## 1. Goals & non-goals

| | |
|---|---|
| **G1** | precision@5 lift на L2-задачах с rag-system-target (91 файл) — измеримый A/B |
| **G2** | Kill-switch: `RAG_RERANKER_ENABLED=false` восстанавливает текущее поведение byte-for-byte |
| **G3** | Graceful fallback — при reranker error (timeout, 5xx) retrieval продолжает с plain HNSW top-k |
| **G4** | Retrieval wall-time overhead ≤ 300ms (reranker 30 docs на qwen-coder-long-sized запрос) |
| **NG1** | Не меняем HNSW параметры (ef, M) — не эта итерация |
| **NG2** | Не добавляем BM25 — v1.34 |
| **NG3** | Ollama backend не поддерживает `/v1/rerank` — не реализуем, Ollama path игнорирует reranker |

## 2. Архитектура

```
было:
  query → embedWithCache(query) → HNSW.search(k=5) → [5 symbols] → context

станет:
  query → embedWithCache(query) → HNSW.search(k=30) → rerank(query, 30 docs) → top-5 → context
                                                          ↑
                                              POST /v1/rerank → llama-swap "reranker"
                                              (если disabled/error → fallback: HNSW top-5)
```

**API shape (подтверждено живым тестом):**
```
POST /v1/rerank
{ "model": "reranker", "query": "...", "documents": ["...", "..."] }
→ { "results": [{ "index": 0, "relevance_score": -5.67 }, ...] }
```
Scores — негативные логиты cross-encoder, ближе к 0 = релевантнее. Сортируем DESC.

## 3. Phases

### Phase A — Config + model-router (~0.5 дня)

**Config** (`packages/shared/src/config.ts`, секция `rag`):
```ts
rerankerEnabled:    envBool('RAG_RERANKER_ENABLED', false),
rerankerModel:      env('LLM_RERANK_MODEL', 'reranker'),
rerankerCandidates: Math.max(1, envInt('RAG_RERANKER_CANDIDATES', 30)),
```
Дефолт `false` — фича opt-in, не ломает текущий деплой.

**ModelBackend** (`backend.ts`):
```ts
rerank?(query: string, documents: string[], model?: string): Promise<RerankResult[]>;
```
```ts
export interface RerankResult { index: number; relevanceScore: number; }
```
Опциональный метод — Ollama не реализует, вызов через `backend.rerank?.(...)`.

**LlamaSwapClient** (`llamaswap-client.ts`):
- `async rerank(query, documents, model?)` → `POST /v1/rerank`
- Возвращает `RerankResult[]` отсортированный DESC по `relevanceScore`
- Бросает при non-ok HTTP (caller делает fallback)

**Tests:** 4 unit-теста на mock fetch:
- success → правильный порядок
- empty documents → `[]`
- HTTP 5xx → throws
- модель override через аргумент

### Phase B — GraphRetriever two-pass (~0.5 дня)

`retrieveContextItems(query, k=5)` в `graph-retriever.ts`:

```
if rerankerEnabled && embedClient.rerank exists:
  candidates = vectorStore.search(queryVec, config.rag.rerankerCandidates)  // 30
  docs = candidates.map(r => truncate(symbol.text, 400 chars))
  try:
    ranked = await embedClient.rerank(query, docs, config.rag.rerankerModel)
    ordered = ranked.slice(0, k).map(r => candidates[r.index])
  catch:
    logger.warn('reranker failed, falling back to HNSW order')
    ordered = candidates.slice(0, k)
else:
  ordered = vectorStore.search(queryVec, k)  // текущее поведение

// дальше — 1-hop deps + token budget, без изменений
```

Truncation 400 символов на документ — компромисс: символ-метод помещается целиком, большой класс режется. Reranker видит достаточно для scoring, batch не распухает.

**Tests:** 4 unit-теста:
- reranker enabled → порядок изменился (mock rerank меняет индексы)
- reranker disabled → вызов `vectorStore.search(k=5)` (не 30)
- reranker throws → fallback к HNSW порядку, warn-лог
- `rerank` не определён на backend (Ollama path) → fallback

## 4. Acceptance criteria

- [ ] AC1 — `npm test` зелёные, +8 новых unit-tests (4 client + 4 retriever)
- [ ] AC2 — `RAG_RERANKER_ENABLED=false` → поведение идентично текущему (нет регрессий)
- [ ] AC3 — L4.1 ×3 ≥ 3/3 commits (как v1.32-c.1 baseline)
- [ ] AC4 — L1.2, L1.3 на sandbox: commit rate ≥ 2/3 (первый baseline, не hard target)
- [ ] AC5 — L2.1, L2.2 на rag-system-target: зафиксирован baseline precision@5 (даже 0% — важно знать)
- [ ] AC6 — на L2.1/L2.2 reranker ≥ baseline (не хуже), lift измерен
- [ ] AC7 — retrieval wall-time overhead ≤ 300ms (измерить через debug-лог)

## 5. Risks & mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| BGE-reranker ctx-size=512 режет длинные символы | MED | Truncation 400 символов; для большинства методов хватает |
| Reranker latency > 300ms при 30 docs | MED | Измерить в Phase C; если >300ms — уменьшить candidates до 15 |
| Scores отрицательные — сортировка DESC правильная? | LOW | Подтверждено тестом: getHealth(-5.67) > createUser(-8.64) ✅ |
| Ollama path ломается | LOW | `rerank?.(...)` — опциональный вызов, Ollama не падает |

## 6. Open questions

- Оптимальный `RAG_RERANKER_CANDIDATES`: 30 — стартовое значение. Если latency ok — можно поднять до 50 в следующей итерации.
- Truncation length: 400 символов — эмпирически, уточним по bench результатам.

## 7. Alternatives considered

- **ONNX reranker в Node.js** (`@xenova/transformers`) — zero-infra, но +150MB npm dep, медленнее на CPU, сложнее деплоить. Отвергнуто — llama-swap path проще и уже работает.
- **Увеличить k HNSW до 10** — не меняет recall, только noise. Отвергнуто — cross-encoder принципиально лучше cosine для re-ranking.

## 8. Out of scope

- BM25 hybrid search — v1.34
- Multi-hop transitive closure — v1.35
- Батч-оптимизация rerank (сейчас один HTTP call на retrieval) — если понадобится
