# Design — v1.39-c Reviewer feedback loop: full-file context on reject

> **Status:** DRAFT — pre-impl. **Date:** 2026-05-14.
> **Order:** идёт после v1.39-b (validation timeout + BUGFIX `_clear`).
> **Links:** [v1.30 tool-calling Coder](v1.30-tool-calling-coder.md) (предок) · [v1.35 pre-Reviewer TS check](v1.35-coder-reviewer-fix.md) (соседний use case BUGFIX_SPEC).

## TL;DR

При `TOOL_CALLING_CODER=true` (default с v1.32-d) **step-level Reviewer-reject путь по-прежнему вызывает patch-based `this.fixer.execute`** — это unintended hybrid из v1.30 эпохи. Patch-Fixer видит `currentChanges` как `{action:'modify', edits:[{search,replace}]}`, то есть **только diff, не full file content** — корень "L2.x reviewer_reject" из v1.38 bench (T6, H4: 2/12 fails ≈ 17%). Фикс: на reject пути переключиться на tool-calling Fixer через `BUGFIX_SPEC`, который умеет `read_file` за полным контентом. Ожидаемый эффект: `reviewer_reject` 29% → ≤15% на real-repo L2/L3.

## 1. Goals & non-goals

| | |
|---|---|
| **G1** | Step-level Reviewer-reject путь использует BUGFIX_SPEC когда `toolCallingCoder=true` (унификация с pre-Reviewer TS check + validation loop) |
| **G2** | Patch-based fallback сохраняется при `TOOL_CALLING_CODER=false` (regression-debug режим) |
| **G3** | Reviewer issues пробрасываются в `BUGFIX_SPEC` через тот же `issues + currentFiles` контракт что validation loop |
| **G4** | Existing L1.x / L4.x sandbox bench не регрессирует (Gemma 4 baseline 4/4) |
| **NG1** | Не меняем Reviewer prompt (это v1.36 lenient Reviewer) — фикс на стороне Fixer-а |
| **NG2** | Не добавляем "explicit final rescue attempt" с pre-rendered full content — это v1.39-d если bench покажет недостаточный эффект |
| **NG3** | Не трогаем validation loop (там уже BUGFIX_SPEC через v1.32-c migration) |

## 2. Архитектура

**Сейчас (orchestrator.ts:704-758, step-level Reviewer loop):**

```
attempt 0..MAX_RETRIES:
  reviewer.execute(stepDesc, currentChanges, reviewContext, mode)
  if approved → break
  else:
    fixer.execute(review.issues, currentChanges, promptContext, mode)
                  ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                  patch-based — видит только edits, нет read_file
    currentChanges = mergeFixerChanges(currentChanges, fixerResult.files)
```

**После (тот же path, dispatch по `config.agents.toolCallingCoder`):**

```
attempt 0..MAX_RETRIES:
  reviewer.execute(...)
  if approved → break
  else:
    fixResult = toolCallingCoder
      ? runTaskAgent(BUGFIX_SPEC, {                     ← новый branch
          stepDescription: step.description,
          context: promptContext,
          taskMode: mode,
          issues: review.issues,                        ← Reviewer text passes through
          currentFiles: currentChanges,
        }, router, writer.root)
      : fixer.execute(review.issues, currentChanges, promptContext, mode)
    currentChanges = mergeFixerChanges(currentChanges, fixResult.files)
```

`BUGFIX_SPEC` уже принимает `{issues, currentFiles}` (validation-mode path) и буде trigger Fixer'у `read_file` любого упомянутого пути → write-scope открывается → edit through `replace_method` / `add_import` / structural tools. Та же машинерия, что сейчас работает в pre-Reviewer TS check и validation.

**Затронутые файлы:**
- [packages/agents/src/orchestrator.ts](../../packages/agents/src/orchestrator.ts) — заменить `fixer.execute(...)` на dispatch (≈12 строк), step-level loop ~704-758.
- [packages/agents/src/__tests__/orchestrator.test.ts](../../packages/agents/src/__tests__/orchestrator.test.ts) — +2 теста: BUGFIX_SPEC вызывается на Reviewer reject когда tool-calling on; patch-Fixer fallback когда off.

Нет изменений в shared / BUGFIX_SPEC / Reviewer prompt.

## 5. Open questions

1. **Need сохранять историю старых Reviewer issues между retry?** Сейчас `runTaskAgent` каждой попытки получает свежий `issues` от текущего Reviewer call'а — никакого "running list of past reject reasons". Tool-calling Fixer внутри одной runTaskAgent'у имеет историю tool-calls, но между Reviewer iterations контекст не переносится. **Решение:** не переносим для v1.39-c — это потенциальный v1.40 enhancement если AC4 не закроется.

2. **maxToolCalls бюджет для BUGFIX_SPEC на Reviewer-reject пути?** Сейчас в spec.ts `maxToolCalls: 30`. Validation loop тоже использует 30. Этого хватает на одну атомарную правку. Не меняем.

3. **Что если Reviewer reject касается test файла, который Coder не туchал?** BUGFIX_SPEC `interceptToolCall` блокирует `create_file` на test paths. Это intentional — на reject Fixer должен **fixить production**, не сочинять новые тесты. Сохраняем.

## 6. Rollback

`TOOL_CALLING_CODER=false` сразу возвращает к v1.38 поведению (patch-Fixer на step-level reject). Откатить без релиза.
