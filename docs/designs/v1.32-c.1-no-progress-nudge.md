# Design — v1.32-c.1 no-progress nudge before done()

## TL;DR

[v1.32-c bench](../benchmarks/runs/2026-05-04-v1.32-c-task-agents.md) surfaced
`done-after-structural-error` pattern in 3/11 runs (L1.1 #3, L4.1 #1, L4.1 #2):
Coder/Fixer hits a structural-tool error (`add_route` / `replace_method` /
`replace_function`), emits `done()` immediately, exits loop with 0 successful
file changes. Pathology guard threshold (5 same-fingerprint errors) doesn't
catch it — model bails after 1 error.

Fix: track `successfulEdits` counter in the shared loop. When `done()` is
called with 0 successful edits, push a "no-progress" strategy nudge (suggest
`replace_in_file` fallback), set `noProgressNudgeFired = true`, continue loop.
After one nudge: allow done() to terminate normally even if still 0 edits
(legitimate give-up). Cap = 1 nudge per loop, no infinite-loop risk.

**Target:** lift L1.1 ×3 to ≥3/3 (from 2/3) and L4.1 ×3 to ≥3/3 (from 3/5)
without regressing healthy runs or wall-time.

## 1. Goals & non-goals

| | |
|---|---|
| **G1** | Block premature `done()` from terminating loop when 0 successful tool effects accumulated; nudge model toward `replace_in_file` fallback. |
| **G2** | Cap nudge at 1/loop (no infinite-nudge risk if model keeps bailing). |
| **G3** | Re-bench L1.1 ×3 + L4.1 ×3; AC met = ≥3/3 each. |
| **NG1** | Не меняем pathology guard logic — orthogonal failure mode (5+ same-fingerprint errors). Both can co-exist in the same loop. |
| **NG2** | Не вводим per-spec nudge customization. Single shared nudge in runner.ts; refactorable later if specs diverge. |
| **NG3** | Не трогаем legacy `ToolCalling{Coder,Fixer}` thin wrappers — изменение в `runTaskAgent` автоматически распространяется через них. |

## 2. Архитектура

### 2.1 Counter logic in `runTaskAgent` (runner.ts)

Two new local variables:
```ts
let successfulEdits = 0;
let noProgressNudgeFired = false;
```

After each tool dispatch:
- `result.text` starts with `error:` → no change (already counted in pathologyStrikes channel)
- `call.function.name === 'read_file'` → no change (read isn't an edit)
- `call.function.name === 'done'` → handled specially (see below)
- Otherwise (any successful write/structural call): `successfulEdits++`

When dispatched call is `done`:
- `result.done = true` (current behavior — `dispatchToolCall` returns `{done: true, text: 'ok: changes finalized'}`)
- New logic — instead of `doneCalled = true; break`:
  ```ts
  if (result.done) {
    if (successfulEdits > 0 || noProgressNudgeFired) {
      doneCalled = true;
      break;
    }
    // 0 successful edits AND nudge not yet fired → intercept done()
    messages.push({ role: 'user', content: NO_PROGRESS_NUDGE });
    noProgressNudgeFired = true;
    logger.info({ agent: spec.agentName }, 'No-progress nudge: blocked premature done()');
    // continue outer round-loop; do NOT break
  }
  ```

### 2.2 Nudge text

Hardcoded const in `runner.ts` (no per-spec customization in v1.32-c.1):

```
You called done() with 0 successful edits — only errors so far. Don't give up
after one structural-tool error. Common recoveries:
- Try replace_in_file with explicit start_line/end_line/new_text on the target
  file (line numbers shift after edits — read_file first to confirm current
  content).
- If add_route errored "no Fastify route calls in file", the route lives in a
  different file (e.g. routes/users.ts) — read that file and add the route there.
- If replace_method errored "class X not found", the target may be a const
  object literal — use replace_in_file on the relevant lines instead.
Only call done() once you have at least one successful tool result.
```

~10 lines, ~600 chars, fits comfortably in prompt-cache delta.

## 5. Risks & mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Nudge causes infinite loops if model keeps calling done() with 0 edits | HIGH if uncapped | `noProgressNudgeFired` flag caps at 1 nudge per loop; second done() exits |
| Nudge text confuses model on legitimate 0-edit cases (e.g. step is "no-op verify") | LOW | Verify via existing test fixtures; if a step truly has no edits, model can call done() once, get nudge, call done() again — adds 1 wasteful round-trip but no functional break |
| Counter reset across rounds breaks tracking | LOW | Locals declared outside outer round-loop, accumulated across rounds (matches `toolCallsExecuted` pattern) |
| Pathology guard interaction — nudge and pathology fire on same call | LOW | Orthogonal channels (consecutiveSameToolErrors vs successfulEdits); both nudges can stack in messages without conflict |
| Per-spec specialization needed later | LOW | Hardcoded const easily refactored to `spec.noProgressNudge()` if specs diverge |

## 6. Open questions

- Should `successfulEdits` count `delete_file` ok? **Decision:** yes — deletion is a real edit effect. Practically rare in failure-mode traces but consistent.
- Should the nudge be info-level log or debug? **Decision:** info — operator-visible signal that the loop intervened, like pathology nudge.

## 7. Alternatives considered

- **Lower pathology threshold to 1.** Rejected — pathology guard fires on *same fingerprint*, not on any error. Lowering would break healthy runs that have 1-2 transient errors.
- **Force `replace_in_file` as first-tool-tried via prompt.** Rejected — undoes the structural-first benefit on tasks where it works (most L3.1, most healthy L1.1).
- **Per-spec nudge text.** Deferred — single text suffices for v1.32-c.1; refactor only if benches show spec-specific guidance helps.

## 8. Out of scope

- Same-call-pattern variants (nudge after 2 errors? after 3?). One-error → nudge is the simplest threshold and matches observed bail patterns.
- Nudge for Validation Fixer's separate concerns (test-vs-production navigation). That's v1.32-d.3 territory.
