# Design — v1.34 Hybrid search (BM25 + dense, RRF)

> **Status:** DRAFT — pre-impl. **Date:** 2026-05-08.
> **Order:** идёт после v1.33 (BGE-reranker).
> **Links:** [v1.33 design](v1.33-reranker.md) · [bench v1.33](../benchmarks/runs/2026-05-07-v1.33-reranker.md)

## TL;DR

Dense-only поиск (HNSW + nomic-embed) не находит `src/server.ts` по запросу "add request-logging middleware" — файл семантически далёк от задачи (не содержит ни `addHook`, ни `durationMs`). BM25 по symbol names + path components даёт точное token-совпадение: `server` → `server.ts`. Добавляем pure-TS BM25 индекс в `GraphRetriever`, сливаем с dense через Reciprocal Rank Fusion (RRF), обёртываем kill-switch `RAG_BM25_ENABLED`. Попутно исключаем `data/backups/**` из индексации (шум в каждом retrieval на target). Ожидаемый эффект: precision@5 на L2.1/L2.2 с 0% → 50–70%.

## 1. Goals & non-goals

| | |
|---|---|
| **G1** | precision@5 > 0/3 на L2.1 (server.ts) и L2.2 (routes/users.ts) — bench подтверждает |
| **G2** | `data/backups/**` исключён из indexCodebase — ни одного backup-файла в retrieval на target |
| **G3** | Kill-switch `RAG_BM25_ENABLED=false` → behaviour идентично v1.33 (регресс-safe) |
| **G4** | Совместимость с reranker: BM25+dense → RRF merge → reranker (если включён) |
| **G5** | Без внешних зависимостей — pure TypeScript BM25 |
| **NG1** | Персистентность BM25 индекса на диск — перестраивается из CodeGraph при `loadFromDisk` |
| **NG2** | Тюнинг k1/b/k по codebase-специфичным данным — используем standard defaults |
| **NG3** | Field-weighted BM25 (отдельные поля с весами) — простое повторение name × 3 вместо |

## 2. Архитектура

```
Query
  │
  ├─ embed(query)           ├─ tokenize(query)
  │       │                 │
  │   VectorStore           BM25Index
  │   .search(vec, 30)      .search(tokens, 30)
  │       │                 │
  │   dense_results[]   bm25_results[]
  │       │                 │
  └───────┴── RRF merge ────┘
                 │
           merged_top_30
                 │
          reranker? → top-5
                 │
           ContextItem[]
```

**RRF formula:** `score(d) = 1/(k + rank_dense) + 1/(k + rank_bm25)`, k=60. Если документ только в одном списке — второй ранг = ∞ (вклад = 0).

**BM25 corpus per symbol:**
```
[...filePath.split('/').flatMap(p => p.split('.')),  // "src", "server", "ts"
 name, name, name,                                    // 3× weight on symbol name
 kind,                                                // "function", "class"
 text]                                                // full body
.join(' ')
```

**Токенизация:** `str.toLowerCase().split(/\W+/).filter(t => t.length >= 2)`

- [packages/rag/src/bm25.ts](../../packages/rag/src/bm25.ts) — `BM25Index` class
- [packages/rag/src/__tests__/bm25.test.ts](../../packages/rag/src/__tests__/bm25.test.ts) — unit tests

**Изменённые файлы:**
- [packages/rag/src/graph-retriever.ts](../../packages/rag/src/graph-retriever.ts) — `BM25Index` поле, `indexFile` строит BM25 + dense, `retrieveContextItems` делает RRF merge, `indexCodebase` исключает backupsPath

## 3. Phases

### Phase A — BM25Index core

Файл: `packages/rag/src/bm25.ts`

```typescript
export class BM25Index {
  private k1 = 1.5;   // TF saturation
  private b  = 0.75;  // length normalization
  private k  = 60;    // RRF constant

  private docs: Map<string, string[]> = new Map();   // id → token array
  private df: Map<string, number> = new Map();        // term → doc count
  private N = 0;
  private totalLen = 0;

  add(id: string, text: string): void
  remove(id: string): void
  search(query: string, topK: number): Array<{ id: string; score: number }>
  rrf(denseIds: string[], bm25Ids: string[], k: number): string[]
  get size(): number
}
```

- `tokenize(text)`: `text.toLowerCase().split(/\W+/).filter(t => t.length >= 2)`
- `add()`: tokenize → обновить `df` для новых термов → сохранить tokens
- `remove()`: уменьшить `df` для каждого терма → удалить из `docs`
- `search()`: BM25 score по запросу, вернуть topK sorted desc
- `rrf()`: принимает два упорядоченных массива id, возвращает RRF-merged список

Tests: 8 unit tests (exact match на name, path-component match, remove, rrf deduplication, empty index, tokenization edge cases).

### Phase B — GraphRetriever integration

**`indexFile` (добавить):**
```typescript
// Build BM25 corpus string per symbol
const corpus = [
  ...symbol.filePath.split('/').flatMap(p => p.split('.')),
  symbol.name, symbol.name, symbol.name,
  symbol.kind,
  symbol.text,
].join(' ');
this.bm25Index.add(symbol.name, corpus);
```

Удалять при `removeFile` → `this.bm25Index.remove(sym.name)`.

**`retrieveContextItems` (изменить):**
```typescript
if (bm25Enabled && this.bm25Index.size > 0) {
  const bm25Results = this.bm25Index.search(query, bm25Candidates);
  const denseIds = candidates.map(c => c.id);
  const bm25Ids  = bm25Results.map(r => r.id);
  const merged   = this.bm25Index.rrf(denseIds, bm25Ids, 60);
  // rebuild results[] from merged order, preserving VectorSearchResult shape
  candidates = merged.map(id => candidates.find(c => c.id === id)
    ?? bm25Results.find(r => r.id === id) && { id, distance: 0 })
    .filter(Boolean);
}
```

**`indexCodebase` — backups exclude:**
```typescript
// Derive absolute backups path and add to ignore list
const absBackups = path.resolve(rootDir, config.safeExec.backupsPath);
const ignorePatterns = [
  ...config.codeGraph.exclude.map(e => `**/${e}/**`),
  path.relative(absRoot, absBackups) + '/**',   // "data/backups/**"
];
```

**`loadFromDisk` — rebuild BM25:**
```typescript
await this.vectorStore.loadFromDisk();
await this.codeGraph.loadFromDisk();
// Rebuild BM25 from in-memory CodeGraph (no separate persistence needed)
for (const symbol of this.codeGraph.getAll()) {
  const corpus = [...].join(' ');
  this.bm25Index.add(symbol.name, corpus);
}
```

**Config additions (`packages/shared/src/config.ts`):**
```typescript
bm25Enabled: envBool('RAG_BM25_ENABLED', true),
bm25Candidates: envInt('RAG_BM25_CANDIDATES', 30),
```

Tests: 2 новых (backups file never appears in retrieval results; BM25 kill-switch preserves v1.33 behaviour).

## 4. Acceptance criteria

- [ ] AC1 — 512+/512 unit tests (все зелёные, BM25 добавляет ≥8 тестов)
- [ ] AC2 — `RAG_BM25_ENABLED=false` → behaviour идентично v1.33 (no regression на L1.x, L4.1)
- [ ] AC3 — L2.1 precision@5 ≥ 1/3 (server.ts попадает в top-5 хотя бы в 1 из 3 прогонов)
- [ ] AC4 — `data/backups/**` файлы не появляются ни в одном retrieval на target
- [ ] AC5 — L1.1/L1.2/L1.3/L4.1 score не хуже v1.33 при `RAG_BM25_ENABLED=true`

## 5. Risks & mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| BM25 шумит на large codebase (91 файл) — все функции match "function" | MED | Stoplist для generic tokens: `function`, `class`, `const`, `export`, `return` (< 3 chars filter уже убирает `if`, `of`) |
| RRF merge порядок может ухудшить качество если BM25 ставит нерелевантный файл на #1 | MED | reranker (если включён) финально переупорядочивает; bench покажет |
| `loadFromDisk` rebuild BM25 медленный для 10K символов | LOW | BM25 add — O(tokens per doc), для 10K символов по 100 токенов = 1M ops ~50ms |
| `bm25Index.remove` неправильно обновляет df → degraded search | MED | Unit test на remove + re-search |

## 6. Open questions

- Нужен ли stoplist для code-specific generic tokens (`function`, `const`, `async`, `return`, `export`)? Ответ после bench — если L1.x деградирует, добавим.
- Нужно ли weight boosting для path components vs body text, или повтор name×3 достаточен? Проверим на bench.

## 7. Alternatives considered

- **sqlite-fts5** — более зрелый FTS, но добавляет native dependency и усложняет setup. Не нужно для 10K символов.
- **Lunr.js / flexsearch** — внешние зависимости, не нужно для нашего объёма.
- **TF-IDF без BM25** — BM25 превосходит TF-IDF по всем публичным бенчмаркам, реализация сопоставима по сложности.

## 8. Out of scope

- v1.35 Multi-hop transitive closure (следующая итерация)
- Field-weighted BM25 с отдельными полями (name/path/body) — pure concat достаточно на этом scale
- Persisting BM25 index to disk (rebuild from CodeGraph — fast enough)
