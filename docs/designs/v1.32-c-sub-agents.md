# Design — v1.32-c Sub-agents (Phase 3 architectural closure)

> **Status:** DRAFT — pre-impl design для критики. Code НЕ trogается до явного approve.
> **Author:** assistant (review by operator)
> **Date:** 2026-05-01
> **Predecessor:** [v1.32-a.5 pathology guard](../benchmarks/runs/2026-04-30-v1.32-a.5-pathology-guard.md)
> **Scope:** ~1 неделя реализации. Phase 3 closure.

## TL;DR

Один универсальный Coder + один универсальный Fixer, разделённые только на «обычный» и
«Fixer» режимы, упёрлись в потолок: в L4.1 (bug fix) Coder не знает что нужно
*навигироваться* от теста к production-модулю, в L3.1 (refactor) дефолт structural-tools
вреден потому что AST-якорей нет, а в L1.1 (feature add) дефолт идеален. Эти три
паттерна различаются только **системным промптом + начальной scope-policy + дефолтным
порядком инструментов**, а не самими примитивами.

v1.32-c вводит три специализированные sub-agent'а — `FeatureAgent`, `BugFixAgent`,
`RefactorAgent` — построенные **поверх существующих tool primitives** (`TOOL_DEFINITIONS`,
`dispatchToolCall`, `WorkingSet`, scope policy, pathology guard). Planner присваивает
`step.kind`, orchestrator маршрутизирует. Patch-based legacy не трогается. Behind
`SUB_AGENTS_ENABLED=true` env flag.

**Что НЕ меняем:** tool primitives, WorkingSet, dispatchToolCall, brace-balance, pathology
guard, no-tool-calls retry, history pruning, Reviewer/Tester/validation loop shape,
patch-based Coder/Fixer.

**Что меняем:** prompt + initial scope + default tool ordering для трёх специализаций;
PlannerAgent выдаёт `step.kind`; orchestrator выбирает sub-agent через registry.

## 1. Goals & non-goals

### Goals

- **G1.** Каждая bench-задача (L1.1 feature, L3.1 refactor, L4.1 bug-fix) маршрутизируется
  на специализированного агента, чьи prompt + scope + tool-order соответствуют
  природе задачи.
- **G2.** Tool primitives shared. Sub-agent — это thin specialization, не parallel
  реализация. ≥90% кода переиспользуется.
- **G3.** Co-existence с patch-based legacy: старый pipeline продолжает работать под
  `TOOL_CALLING_CODER=false` и тестируется в обоих режимах. v1.32-c — третий путь
  (`SUB_AGENTS_ENABLED=true`), а не замена.
- **G4.** Валидируется на `L1.1 + L3.1 + L4.1` benches: каждый landed-commit, без
  регрессий vs v1.32-a.5 baseline.
- **G5.** Bug-fix workflow получает explicit "trace from test to production" hint в
  prompt'е BugFixAgent — это устраняет gap, который v1.32-a.x закрывал косвенно
  через scope discipline.

### Non-goals

- **NG1.** НЕ вводить четвёртого sub-agent'а в этой итерации (MigrationAgent —
  отложен до bench evidence что v1.32-c работает).
- **NG2.** НЕ переписывать TOOL_DEFINITIONS. Они стабильны и хорошо тестированы.
- **NG3.** НЕ улучшать model reasoning (это Phase 4, llama.cpp + grammar-constrained
  tool calls).
- **NG4.** НЕ вводить sub-agent-specific tool sets (e.g. RefactorAgent видит всё, но
  предпочитает `replace_in_file`). Это проще решается через prompt'ы и default
  ordering, чем через фильтрацию schema'ы.
- **NG5.** НЕ менять Reviewer / Tester / validation loop. Они работают на
  `currentChanges: FileChange[]` и не зависят от того, какой агент их произвёл.

## 2. Baseline (что есть сейчас)

### 2.1 Текущая иерархия агентов

```
PlannerAgent      → выдаёт PlanStep[] {id, description, dependencies}
ArchitectAgent    → per-step design pre-pass (design: string)
CoderAgent        ← legacy patch-based (TOOL_CALLING_CODER=false)
ToolCallingCoderAgent  ← v1.30+, default
TesterAgent       → генерирует test files
ReviewerAgent     → approves/rejects step output
FixerAgent        ← legacy patch-based
ToolCallingFixerAgent  ← v1.30.3+, default
```

### 2.2 Текущая маршрутизация

`Orchestrator.executeStep` (orchestrator.ts:514):

```ts
const codeChanges = config.agents.toolCallingCoder
  ? await this.toolCallingCoder.execute(step.description, ...)
  : await this.coder.execute(step.description, ...);
```

То есть **один Coder на всё** — feature-add, refactor, bug-fix не различаются.
Аналогично `runValidationLoop` (orchestrator.ts:729) — один Fixer.

### 2.3 Какие сигналы уже есть

- **Pathology guard** (`PATHOLOGY_THRESHOLD = 5`, `MAX_PATHOLOGY_STRIKES = 2`) — общий,
  shared между Coder и Fixer через imports из `tool-calling-coder.ts`.
- **No-tool-calls retry** (2 retries с прогрессивно сильными nudges) — общий шаблон.
- **History pruning** — only Fixer (`HISTORY_PRUNE_THRESHOLD = 22`); Coder не нужен,
  потому что MAX_TOOL_CALLS = 50 ≪ pruning threshold для типичной длины.
- **Scope policy** (`WritePolicy { allowed, forbiddenPatterns }`) + read-grants-write
  (`isWriteAllowed(path, policy, ws)`) — общий механизм. Дифференциация:
  - Coder: `allowed = extractAllowedPaths(stepDescription)`, `forbidden = ALWAYS_FORBIDDEN_PATTERNS`
  - Fixer: `allowed = buildFixerAllowedSet(coderFiles, issues)` (с тестами отфильтрованными),
    `forbidden = ALWAYS_FORBIDDEN_PATTERNS + FIXER_TEST_PATH_FORBIDDEN`

### 2.4 Что отличает три класса задач (наблюдаемо в L1.1 / L3.1 / L4.1)

| | L1.1 feature add | L3.1 refactor | L4.1 bug-fix |
|---|---|---|---|
| Файл назван в task'е | yes — explicit | partially — describes a transformation | implicit — описывает симптом |
| Структурный tool fits | yes — `add_route` / `add_method` | no — `replace_in_file` (object-literal) | depends — `replace_method` если method, else replace_in_file |
| Test failure роль | в Tester'е (генерируется свежий) | как validation gate | как **навигационный сигнал** к bug |
| Prompt должен подсказать | "use structural" | "AST tools won't fit, fallback OK" | "trace from test to production module" |
| Scope expansion typical | small — пара файлов | medium — refactor target | larger — production behind tests |

**Это и есть три специализации**, которые v1.32-c кодирует.

## 3. Архитектура: что общее, что специализируется

### 3.1 Shared core (existing — не меняем)

```
TOOL_DEFINITIONS              — 11 tool schemas; экспортируется из tool-calling-coder.ts
dispatchToolCall(call, ws, p) — единый dispatcher; не sub-agent-aware
WorkingSet                    — virtual filesystem
checkBraceBalance             — defense-in-depth syntax check
isWriteAllowed                — scope policy enforcement
extractAllowedPaths           — task-description path extraction
ALWAYS_FORBIDDEN_PATTERNS     — package.json, tsconfig, etc.
PATHOLOGY_THRESHOLD = 5
MAX_PATHOLOGY_STRIKES = 2
FILE_ARG_KEY                  — tool_name → arg_key map
pruneHistory                  — Fixer history compaction
```

### 3.2 Что специализируется per sub-agent

```
SYSTEM_PROMPT                 — workflow + tool-ordering hints
buildAllowedSet(...)          — initial scope construction
extraForbiddenPatterns        — sub-agent-specific bans (test-path forbidden = BugFixAgent only)
MAX_TOOL_CALLS                — Feature 50, BugFix 30, Refactor 40 (см. §4)
nudgeFlavor                   — strategy nudge для pathology guard может ссылаться на разные tool subsets
```

### 3.3 Предлагаемая class hierarchy

```
abstract class ToolCallingAgentBase {
  // Общие constants + helper'ы; в нём живёт execute()-loop с pathology guard +
  // no-tool-calls retry + (опционально) history pruning. Configured через protected
  // методы которые subclass переопределяет.

  protected abstract systemPrompt: string;
  protected abstract maxToolCalls: number;
  protected abstract pruneHistoryEnabled: boolean;
  protected abstract buildAllowedSet(input: AgentInput): Set<string>;
  protected abstract buildForbiddenPatterns(): RegExp[];
  protected abstract buildUserMessage(input: AgentInput, scopeLine: string): string;
  protected abstract pathologyNudgeFor(tool: string, path: string): string;

  async execute(input: AgentInput, projectRoot: string): Promise<{ files: FileChange[] }> {
    // shared loop: build policy, run rounds, dispatch, track pathology, return ws.toFileChanges()
  }
}

class FeatureAgent  extends ToolCallingAgentBase { ... }
class BugFixAgent   extends ToolCallingAgentBase { ... }
class RefactorAgent extends ToolCallingAgentBase { ... }
```

`ToolCallingCoderAgent` и `ToolCallingFixerAgent` остаются (для legacy paths и pre-v1.32-c
tests). Они могут быть **переписаны** как:

```ts
class ToolCallingCoderAgent extends FeatureAgent {}    // alias, default behavior
class ToolCallingFixerAgent extends BugFixAgent {}     // alias, current Fixer behavior
```

— или оставлены отдельно, чтобы не риск регресса. Решение: **оставить ToolCalling{Coder,Fixer}
без изменений** в первом коммите; sub-agents — параллельная реализация. Удалим дубль
во второй итерации, когда sub-agents проверены bench'ами.

### 3.4 Альтернатива, которую отвергли: tool subsets per sub-agent

Можно было бы отфильтровать `TOOL_DEFINITIONS` per agent (RefactorAgent видит только
`read_file`, `replace_in_file`, `done`; FeatureAgent — full set). Отвергнуто потому что:

- Fragmentation тестов: каждый subset нужно тестировать отдельно.
- Модель достаточно умна, чтобы сама выбрать tool по prompt'у — структурный tool на
  refactor задаче уже сегодня корректно отвергается с error'ом ("class X not found"),
  и модель fallback'ает на replace_in_file (наблюдаемо в v1.31.2 L3.1, 3 calls).
- Гибкость: refactor задача, которая ВНЕЗАПНО оказалась AST-anchored, не блокирована.

Tool subsets — это policy-через-schema, prompt'ы — это policy-через-instructions. Второе
дешевле и эквивалентно при достаточном модельном понимании.

## 4. Inventory: три sub-agent'a

### 4.1 FeatureAgent (default; replaces vanilla Coder for additive tasks)

**Когда вызывается:** `step.kind === 'feature'` (Planner default).

**Behavior:**
- `MAX_TOOL_CALLS = 50`
- `pruneHistoryEnabled = false` (короткие сессии, без необходимости)
- `buildAllowedSet`: `extractAllowedPaths(stepDescription)` (current Coder behavior)
- `buildForbiddenPatterns`: `ALWAYS_FORBIDDEN_PATTERNS`
- **System prompt:** current Coder prompt (он уже структурно настроен на additive
  tasks). Минор edits: убираем `Test files are NOT your responsibility` строку
  потому что step.kind=feature всё равно не пишет в тесты в норме.

**Output:** `FileChange[]` — feeds Tester → Reviewer → validation как сейчас.

### 4.2 BugFixAgent (replaces ToolCallingFixer + handles step.kind='bugfix')

**Когда вызывается:** ДВА путя:
- (a) Validation Fixer — `runValidationLoop` invokes BugFixAgent для test/typecheck
  failures (current Fixer role).
- (b) Planner-driven — `step.kind === 'bugfix'`, выполняется в `executeStep` вместо
  FeatureAgent. Описание шага — это симптом ("UserService.create forgets createdAt");
  агент сам навигируется.

**Behavior:**
- `MAX_TOOL_CALLS = 30` (был 25 у Fixer; чуть выше для (b)-path где
  "найди bug" дороже чем "исправь известный issue")
- `pruneHistoryEnabled = true` (длинные сессии — current Fixer pattern)
- `buildAllowedSet`:
  - (a) — `buildFixerAllowedSet(coderFiles, issues)` (current Fixer behavior)
  - (b) — `extractAllowedPaths(stepDescription)` UNION (если step упоминает test path)
    paths упомянутые в task description, **но** test paths отфильтрованы как у Fixer
- `buildForbiddenPatterns`: `ALWAYS_FORBIDDEN_PATTERNS + FIXER_TEST_PATH_FORBIDDEN`
- **System prompt:** current FIXER_SYSTEM_PROMPT + новая секция:
  ```
  NAVIGATION FOR BUG FIXES:
  - Если task упоминает поведение, не файл — начинай с RAG-suggested entry points
    (server.ts, routes/, services/) и trace via read_file.
  - Test failure → bug в production module, который тест exercises. Открой
    тестовый импорт, читай, фикси production.
  - НЕ редактируй test чтобы silence assertion. Test forbidden_pattern enforce'ит это.
  ```

**Output:** `FileChange[]` — same shape.

### 4.3 RefactorAgent (new; step.kind='refactor')

**Когда вызывается:** `step.kind === 'refactor'`.

**Behavior:**
- `MAX_TOOL_CALLS = 40` (refactor может trogать много мест в одном файле)
- `pruneHistoryEnabled = false`
- `buildAllowedSet`: same as FeatureAgent
- `buildForbiddenPatterns`: `ALWAYS_FORBIDDEN_PATTERNS`
- **System prompt:** consolidated из Coder prompt + новая преамбула:
  ```
  REFACTOR DEFAULT TOOL ORDERING:
  - Refactor задачи часто меняют существующий код, который НЕ совпадает с AST
    primitives (e.g. const-object-literal, не FunctionDeclaration). НЕ пытайся
    structural-tool-first — он отвергнется.
  - Workflow: read_file → plan edits → replace_in_file для line-coord changes;
    structural-tool ТОЛЬКО когда символ явно соответствует (replace_method для
    class method, replace_function для FunctionDeclaration).
  - Refactor preserves behavior. Если test проваливается после твоих edits — твой
    refactor что-то сломал, не "test устарел".
  ```

**Output:** `FileChange[]`.

### 4.4 Что общее в prompt'ах (стоит вынести в shared base)

- "tool calls only — text replies do nothing"
- read-grants-write rule (с пояснением)
- ALWAYS_FORBIDDEN list mention (package.json/tsconfig/etc.)
- "line numbers shift after every edit, re-read after mutation"
- "follow project conventions: NodeNext .js suffixes, strict mode, indent style"

→ Вынесем в `SHARED_PROMPT_CORE` константу в `agents/shared-prompts.ts`. Каждый
sub-agent конкатит `SHARED_PROMPT_CORE + SPECIALIZATION` в своём `systemPrompt` getter.

## 5. Planner integration: step.kind

### 5.1 Schema change

```ts
// packages/agents/src/planner.ts
export const PlanStepSchema = z.object({
  id: z.string(),
  description: z.string(),
  dependencies: z.array(z.string()).default([]),
  // NEW:
  kind: z.enum(['feature', 'bugfix', 'refactor']).default('feature'),
});
```

`.default('feature')` обеспечивает backwards-compat — старые planner'ы / fixtures без
`kind` парсятся в feature.

### 5.2 Planner prompt update

Дополнить Planner SYSTEM_PROMPT правилом 7:

```
7. Each step has a `kind`: "feature" | "bugfix" | "refactor". Default "feature".
   - "bugfix": task describes a SYMPTOM ("users see duplicates", "createdAt missing
     in response", "tests fail with X"). Bug location is implicit.
   - "refactor": task describes a TRANSFORMATION of existing code that preserves
     behavior ("convert const-object-literal X to a class", "rename Y to Z",
     "extract function W"). Tests should keep passing.
   - "feature": task describes ADDING new behavior ("add /version endpoint",
     "add soft-delete to users"). Net-new code dominates.
   When in doubt — feature.
```

И в JSON output schema example добавить `"kind": "feature"`.

### 5.3 Альтернатива: классификатор-эвристика вместо planner-driven

**Вариант A:** Planner LLM решает (предложено).
**Вариант B:** Heuristic classifier поверх description: ключевые слова `fix|bug|broken|fails` → bugfix; `refactor|rename|extract|convert` → refactor; иначе feature.

**Решение:** **Вариант A с heuristic fallback.** Planner LLM может выдать невалидный kind
или забыть поле — Zod default'ит в feature; затем мы прогоняем description через
heuristic, и если heuristic сильно не согласен (`bugfix`-keywords при `feature` kind),
логируем warning + override. Это даёт LLM driver + safety net.

```ts
// packages/agents/src/planner.ts (new helper)
export function inferStepKind(description: string): PlanStep['kind'] {
  const d = description.toLowerCase();
  if (/\b(fix|bug|broken|fails?|failing|incorrect|wrong|forgets?)\b/.test(d)) return 'bugfix';
  if (/\b(refactor|rename|extract|convert|migrate|reorganize)\b/.test(d)) return 'refactor';
  return 'feature';
}

// applied as override-if-disagree, not blind replace
```

## 6. Orchestrator integration

### 6.1 Registry

```ts
// packages/agents/src/sub-agents/registry.ts (new file)
import type { PlanStep } from '../planner.js';
import { FeatureAgent } from './feature-agent.js';
import { BugFixAgent } from './bugfix-agent.js';
import { RefactorAgent } from './refactor-agent.js';

export interface SubAgent {
  execute(input: SubAgentInput, projectRoot: string): Promise<{ files: FileChange[] }>;
}

export function pickAgentForStep(
  kind: PlanStep['kind'],
  router: ModelRouter,
): SubAgent {
  switch (kind) {
    case 'bugfix':   return new BugFixAgent(router);
    case 'refactor': return new RefactorAgent(router);
    case 'feature':
    default:         return new FeatureAgent(router);
  }
}
```

### 6.2 executeStep dispatch

```ts
// orchestrator.ts (excerpt)
const codeChanges = config.agents.subAgents
  ? await pickAgentForStep(step.kind, this.router).execute({
      stepDescription: step.description,
      context: promptContext,
      taskMode: mode,
      onFileReady: onCoderFile,
    }, this.writer.root)
  : config.agents.toolCallingCoder
    ? await this.toolCallingCoder.execute(step.description, promptContext, mode, this.writer.root, onCoderFile)
    : await this.coder.execute(step.description, promptContext, mode, onCoderFile);
```

### 6.3 runValidationLoop dispatch

Validation Fixer — всегда BugFixAgent в (a)-mode, **независимо от step.kind** оригинального
шага. Семантика: "в коде что-то сломано, найди и исправь" — это всегда bugfix workflow.

```ts
const fixResult = config.agents.subAgents
  ? await new BugFixAgent(this.router).execute({
      issues, currentFiles: allFileChanges, context: validationContext, taskMode: mode,
    }, this.writer.root)
  : config.agents.toolCallingCoder
    ? await this.toolCallingFixer.execute(issues, allFileChanges, validationContext, mode, this.writer.root)
    : await this.fixer.execute(issues, allFileChanges, validationContext, mode);
```

### 6.4 Config

```ts
// packages/shared/src/config.ts (delta)
export const config = {
```

## 7. Migration / rollout

### 7.1 Phases

**Phase A — landing the scaffolding (~2-3 дня):**
1. `agents/shared-prompts.ts` — `SHARED_PROMPT_CORE` constant
2. `agents/sub-agents/base.ts` — `ToolCallingAgentBase` abstract class с execute() loop
3. `agents/sub-agents/feature-agent.ts`, `bugfix-agent.ts`, `refactor-agent.ts`
4. `agents/sub-agents/registry.ts` + `pickAgentForStep`
5. `planner.ts` — `kind` field + prompt update + `inferStepKind` helper
6. `orchestrator.ts` — dispatch behind `config.agents.subAgents` flag
7. **Tests** (~25-30 unit-tests; см. §8)

**Phase B — bench validation (~1 день):**
- L1.1 ×3 sandbox (FeatureAgent path)
- L3.1 ×3 sandbox (RefactorAgent path)
- L4.1 ×5 sandbox (BugFixAgent path) — robustness vs v1.32-a.5 baseline
- Run-file `2026-05-XX-v1.32-c-sub-agents.md`

**Phase C — cleanup (~0.5 дня, опционально):**
- Если bench зелёный по всем трём kinds: replace `ToolCallingCoderAgent` с `class
  extends FeatureAgent {}` + `ToolCallingFixerAgent` с `class extends BugFixAgent {}`,
  удалить дублирующий код. Это можно отложить на v1.32-c.1.

### 7.2 Backwards compat

- `SUB_AGENTS_ENABLED` default false → новый код dormant до явного opt-in.
- `ToolCalling{Coder,Fixer}Agent` остаются работать (default path).
- `CoderAgent` / `FixerAgent` (patch-based) остаются работать (`TOOL_CALLING_CODER=false`).
- Все 392 unit-теста должны остаться зелёными.

### 7.3 Что комитим в одном PR vs split

**Один PR** — sub-agents как unit. Тестировать atomically. Если bench выявит regression
— rollback одной revert'ой. Plan steps можно разбить на коммиты внутри PR'а:
- commit 1: shared prompts + base class + tests
- commit 2: FeatureAgent + tests
- commit 3: BugFixAgent + tests
- commit 4: RefactorAgent + tests
- commit 5: Planner kind + dispatch + tests
- commit 6: bench run-file

## 10. Risks & open questions

### 10.1 Risks

- **R1 — Planner classifies kind incorrectly.** L4.1-style bug-fix задача,
  classified as feature, попадёт в FeatureAgent, который не знает trace-to-production
  trick. Mitigation: heuristic fallback (§5.3). Severity: MEDIUM (в худшем случае ведём
  себя как v1.32-a.5).
- **R2 — Prompt drift across three agents.** Future maintenance: один и тот же tweak
  нужно реплицировать в три prompt'a. Mitigation: SHARED_PROMPT_CORE + только
  per-agent specialization section. Severity: LOW.
- **R3 — Backwards-compat: existing planner outputs без `kind` field.**
  Mitigation: `.default('feature')` в Zod schema. Severity: LOW.
- **R4 — Bench variance on small sample.** L4.1 ×5 даёт σ-bound ≈40%; одна неудача
  не означает regression. Mitigation: повторить ×5 вторым прогоном если есть doubt.
  Severity: LOW.
- **R5 — Increased Ollama load (3 prompt cache slots вместо 2).** Каждый sub-agent
  имеет свой system prompt; cache hits ниже. Mitigation: SHARED_PROMPT_CORE — большая
  часть prompt'a одинакова, cache'ируется. Severity: LOW.
- **R6 — Class hierarchy refactor легко скейлит к over-engineering.** Соблазн
  добавить hooks, plugins, factories. Mitigation: explicit design constraint — три
  abstract methods, ничего больше. Severity: LOW (если author соблюдает).

### 10.2 Open questions (request operator decision)

- **Q1.** Оставить `ToolCalling{Coder,Fixer}Agent` живыми постоянно (third path) или
  заменить на subclasses в Phase C?
  - Ответ A: live forever — overhead дублирования, но zero risk регресса
  - Ответ B: replace во второй итерации — clean, less code
  - **Default:** A (минимум risk на v1.32-c, B — отложить на v1.32-c.1)
- **Q2.** ArchitectAgent сейчас pre-pass перед Coder. Сохраняем для всех трёх kinds или
  пропускаем для bugfix (where description = symptom, не design)?
  - **Default:** сохраняем для всех — Architect output короткий, не вредит, и BugFix
    может полезно прочитать "design" про подход к navigation.
- **Q3.** TesterAgent generates test files — нужен ли он для refactor (тесты должны
  сохраниться, не создаваться)? И для bugfix?
  - **Default:** disable Tester для refactor + bugfix kinds. Они не "вводят новое
    поведение", тесты уже существуют. Это economy — экономит 1 LLM call на step.
- **Q4.** Имена sub-agent'ов — `FeatureAgent / BugFixAgent / RefactorAgent` или
  `Feature / BugFix / Refactor` (без Agent suffix)?
  - **Default:** с Agent suffix — consistency с PlannerAgent / ArchitectAgent / etc.
