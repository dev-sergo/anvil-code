# Design — v1.32-c task-agents (Phase 3 closure)

## TL;DR

Один Coder + один Fixer упёрлись в потолок: L4.1 (bug-fix) — нет navigation hint, L3.1 (refactor) — structural-tools default вреден, L1.1 (feature) — ок. Различие — **prompt + scope + tool-order**, не примитивы.

v1.32-c вводит три **specs** (`FEATURE_SPEC`, `BUGFIX_SPEC`, `REFACTOR_SPEC`) над shared `runTaskAgent(spec, input, ...)`. Planner присваивает `step.kind`, orchestrator маршрутизирует. Tool primitives (`TOOL_DEFINITIONS`, `dispatchToolCall`, `WorkingSet`, scope policy, pathology guard, brace-balance) НЕ трогаются. ToolCalling{Coder,Fixer} переписываются как тонкие обёртки → удаляется дублирование.

**Path:** `packages/agents/src/task-agents/` (избегаем clash с Claude Code «sub-agents»).

---

## 1. Architecture: flat composition

### 1.1 Spec interface

```ts
// task-agents/spec.ts
export interface TaskAgentSpec {
  kind: 'feature' | 'bugfix' | 'refactor';
  systemPrompt: string;             // built from SHARED_PROMPT_CORE + specialization
  maxToolCalls: number;
  pruneHistory: boolean;
  buildAllowedSet: (input: TaskAgentInput) => Set<string>;
  forbiddenPatterns: RegExp[];
  pathologyNudge: (tool: string, path: string) => string;
}

export interface TaskAgentInput {
  stepDescription: string;
  context: string;
  taskMode: TaskMode;
  // bugfix-only (validation mode):
  issues?: string[];
  currentFiles?: FileChange[];
  onFileReady?: (f: FileChange) => void;
}
```

### 1.2 Shared loop

```ts
// task-agents/runner.ts
export async function runTaskAgent(
  spec: TaskAgentSpec,
  input: TaskAgentInput,
  router: ModelRouter,
  projectRoot: string,
): Promise<{ files: FileChange[] }> {
  // 1. build WritePolicy { allowed: spec.buildAllowedSet(input), forbidden: spec.forbiddenPatterns }
  // 2. construct messages [system: spec.systemPrompt, user: build(input)]
  // 3. loop ≤ spec.maxToolCalls: dispatchToolCall + pathology guard + no-tool-calls retry
  // 4. if spec.pruneHistory: invoke pruneHistory each round
  // 5. return ws.toFileChanges()
}
```

### 1.3 Three specs

| Field | FEATURE_SPEC | BUGFIX_SPEC | REFACTOR_SPEC |
|---|---|---|---|
| `kind` | feature | bugfix | refactor |
| `maxToolCalls` | 50 | 30 | 40 |
| `pruneHistory` | false | true | false |
| `buildAllowedSet` | `extractAllowedPaths(desc)` | (a) `buildFixerAllowedSet` / (b) `extractAllowedPaths` | `extractAllowedPaths(desc)` |
| `forbiddenPatterns` | `ALWAYS_FORBIDDEN` | `ALWAYS_FORBIDDEN + TEST_PATH_FORBIDDEN` | `ALWAYS_FORBIDDEN` |
| Prompt suffix | structural-first hint | "trace test→production" + "no test silencing" | "AST may not fit; replace_in_file fallback OK" |

### 1.4 Shared prompt core

`task-agents/shared-prompts.ts` — `SHARED_PROMPT_CORE` const содержит:
- "tool calls only — text replies do nothing"
- read-grants-write rule
- `ALWAYS_FORBIDDEN` paths mention
- "line numbers shift after edits, re-read"
- project conventions (NodeNext .js suffixes, strict mode)

Каждый spec: `systemPrompt = SHARED_PROMPT_CORE + SPECIALIZATION`. Большая часть кешируется prompt-cache'ом llama-swap.

### 1.5 Registry

```ts
// task-agents/registry.ts
export const SPECS: Record<PlanStep['kind'], TaskAgentSpec> = {
  feature: FEATURE_SPEC,
  bugfix: BUGFIX_SPEC,
  refactor: REFACTOR_SPEC,
};
export const pickSpec = (kind: PlanStep['kind'] | undefined) => SPECS[kind ?? 'feature'];
```

### 1.6 Что НЕ трогаем

`TOOL_DEFINITIONS`, `dispatchToolCall`, `WorkingSet`, `checkBraceBalance`, `isWriteAllowed`, `extractAllowedPaths`, `ALWAYS_FORBIDDEN_PATTERNS`, `PATHOLOGY_THRESHOLD`, `MAX_PATHOLOGY_STRIKES`, `FILE_ARG_KEY`, `pruneHistory`, Reviewer, validation loop shape, patch-based Coder/Fixer.

## 2. Planner integration

### 2.1 Schema

```ts
export const PlanStepSchema = z.object({
  id: z.string(), description: z.string(), dependencies: z.array(z.string()).default([]),
  kind: z.enum(['feature', 'bugfix', 'refactor']).default('feature'),
});
```

`.default('feature')` — BC для старых planner outputs / fixtures.

### 2.2 Prompt rule (добавить как пункт 7)

```
7. Each step has `kind`: "feature" | "bugfix" | "refactor". Default "feature".
   - "bugfix": task = symptom (location implicit, "users see duplicates")
   - "refactor": transformation preserving behavior ("rename X to Y")
   - "feature": new behavior added ("add /version endpoint")
   When in doubt — feature.
```

### 2.3 Heuristic — informational, не override

```ts
export function inferStepKind(description: string): PlanStep['kind'] {
  const d = description.toLowerCase();
  if (/\b(fix|bug|broken|fails?|failing|incorrect|wrong)\b/.test(d)) return 'bugfix';
  if (/\b(refactor|rename|extract|convert|migrate|reorganize)\b/.test(d)) return 'refactor';
  return 'feature';
}
```

В orchestrator: если `inferStepKind(desc) !== step.kind` → `logger.warn({ planned: step.kind, inferred, desc })`. **Не override** — Planner LLM источник истины, heuristic = signal оперaтору что классификация подозрительна.

## 3. Orchestrator dispatch

### 3.1 executeStep

```ts
const codeChanges = await runTaskAgent(
  pickSpec(step.kind),
  { stepDescription: step.description, context: promptContext, taskMode: mode, onFileReady: onCoderFile },
  this.router, this.writer.root,
);
```

(Под `TOOL_CALLING_CODER=false` — patch-based legacy `coder.execute(...)` остаётся.)

### 3.2 runValidationLoop

Validation Fixer — **всегда** `BUGFIX_SPEC` независимо от `step.kind`. Семантика «test failure → bug в production» = bugfix workflow.

```ts
const fixResult = await runTaskAgent(
  BUGFIX_SPEC,
  { stepDescription: '<validation>', context: validationContext, taskMode: mode,
    issues, currentFiles: allFileChanges },
  this.router, this.writer.root,
);
```

### 3.3 Architect / Tester по kind

- **Architect:** skip для `refactor` (transformation = design). Keep для feature/bugfix.
- **Tester:** skip для `refactor` (тесты сохраняются, не создаются). Keep для feature и bugfix-(b) planner-driven (regression test полезен). Validation-mode (a) — Tester и так не вызывается.

### 3.4 Config

`SUB_AGENTS_ENABLED` flag **не вводится.** Default kind = 'feature' даёт byte-identical behavior к текущему `ToolCallingCoderAgent` — это валидируется regression-тестом + bench L1.1 ×1 (AC6). Если красное — flip к старому пути одной revert'ой.

## 4. Phase C cleanup (в этом же PR)

После того как 3 specs работают:

```ts
// tool-calling-coder.ts (тонкая обёртка для callers)
export class ToolCallingCoderAgent {
  constructor(private router: ModelRouter) {}
  execute(desc, ctx, mode, root, onFile) {
    return runTaskAgent(FEATURE_SPEC, { stepDescription: desc, context: ctx, taskMode: mode, onFileReady: onFile }, this.router, root);
  }
}
// tool-calling-fixer.ts — аналогично над BUGFIX_SPEC
```

Удаляется ~600 строк дублирующего loop кода (двух copies pathology guard / no-tool-calls retry / dispatch driver). Все existing 392 тестов на ToolCalling{Coder,Fixer} остаются зелёными — публичный API сохранён.

## 11. Decisions log (revision 2 vs revision 1)

| Q | Rev1 default | Rev2 (this) | Why |
|---|---|---|---|
| Architecture | abstract base + 3 subclasses | flat composition: spec + `runTaskAgent` | Нет state в classes; 7 hooks → 7 fields; параметризация тестов проще |
| Naming | `sub-agents/` | `task-agents/` | Avoid clash с Claude Code «subagents» concept |
| Q1 Phase C cleanup | A: live forever | **B: в том же PR** | Один путь, один тест; revert одной commit'ой |
| Q2 Architect | keep all | **skip refactor** | Transformation = design |
| Q3 Tester | skip refactor+bugfix | **skip refactor; keep bugfix-(b)** | Regression test для planner-driven bugfix |
| Q4 Naming suffix | `*Agent` | n/a — теперь spec const'ы | Functions, не classes |
| Q5 Default flag | opt-in | **no flag** | Default kind=feature byte-identical; regression guard через AC6 |
