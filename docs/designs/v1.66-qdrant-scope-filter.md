# Design — v1.66 qdrant-scope-filter

> **Status:** DRAFT — pre-impl. **Date:** 2026-05-26.
> **Order:** идёт после v1.65d (Phase 4 closed). Открывает Phase 5.
> **Links:** [Phase 5 roadmap](../../ROADMAP.md#phase-5--multi-hop-retrieval--production-storage)

## TL;DR

`VECTOR_BACKEND=qdrant` включён, Qdrant запущен и заполнен (2 коллекции, ~2.3K points). Однако scope-фильтрация сломана: `{ filePath: "packages/openapi" }` уходит в Qdrant как exact match, но payload хранит полный абсолютный путь `/Users/.../packages/openapi/src/foo.ts` — совпадений ноль. Фикс: добавить `packageName` в payload при индексации, переключить фильтр с path-prefix на exact `packageName`. После фикса cross-package context overflow для monorepo-задач типа trpc должен снизиться. Ожидаемый outcome: bench ≥ v1.65d baseline (11/12), scope filter реально срабатывает (верифицируем логами).

## 1. Goals & non-goals

| | |
|---|---|
| **G1** | Scope filter реально срабатывает для Qdrant — верифицировать логами что `packageName` filter исключает символы из других пакетов |
| **G2** | Bench на trpc+hono ≥ v1.65d baseline (11/12) при `VECTOR_BACKEND=qdrant` |
| **G3** | Существующие коллекции мигрированы (payload обновлён с `packageName`) без полного re-index |
| **NG1** | Смена алгоритма retrieval — только payload структура и filter logic |
| **NG2** | Qdrant cloud / distributed setup |
| **NG3** | `packageName`-фильтрация в BM25 (в этой итерации) |

## 2. Архитектура

**Проблема сейчас:**
```
indexFile() → vectorStore.add(name, vec, { filePath: "/abs/path/packages/openapi/src/foo.ts", kind })
                                              ↑ полный путь
retrieveContextItems() → extractPackageScope("...packages/openapi...") → "packages/openapi"
                      → search(vec, k, { filePath: "packages/openapi" })
                                              ↑ НЕ совпадает → фильтр no-op
```

**После фикса:**
```
indexFile() → extractPackageName(filePath) → "openapi"   (или "@trpc/openapi")
           → vectorStore.add(name, vec, { filePath, kind, packageName: "openapi" })

retrieveContextItems() → extractPackageName(query) → "openapi"
                      → search(vec, k, { packageName: "openapi" })
                                              ↑ exact match работает
```

**Функция `extractPackageName()`:**
- Ищет `packages/<name>` в пути/запросе, возвращает `<name>` (только имя пакета, не путь)
- Если путь не содержит `packages/` — возвращает `undefined` (без фильтра)
- Примеры: `"/abs/packages/openapi/src/foo.ts"` → `"openapi"`, `"packages/server/src"` → `"server"`

**Затронутые файлы:**
- [packages/rag/src/qdrant-vector-store.ts](../../packages/rag/src/qdrant-vector-store.ts) — расширить тип filter: `packageName?: string`; добавить в Qdrant must-clause
- [packages/rag/src/graph-retriever.ts](../../packages/rag/src/graph-retriever.ts) — `indexFile()`: добавить `packageName` в metadata; `extractPackageScope()` → `extractPackageName()` (возвращает имя, не путь); `vectorFilter` переключить на `{ packageName }`
- [packages/rag/src/vector-store.ts](../../packages/rag/src/vector-store.ts) — обновить сигнатуру `search()` filter: добавить `packageName?: string` (no-op, для interface compatibility)
- **Новый файл:** `packages/rag/src/scripts/patch-qdrant-payload.ts` — обновить payload у существующих points: читать `filePath` из payload → вычислить `packageName` → Qdrant `set_payload` на каждый point

## 3. Phases

### Phase A — fix indexing payload

В `graph-retriever.ts:414`:
```ts
// было:
await this.vectorStore.add(symbol.name, vector, { filePath, kind: symbol.kind });
// стало:
const packageName = this.extractPackageName(filePath);
await this.vectorStore.add(symbol.name, vector, {
  filePath, kind: symbol.kind, ...(packageName ? { packageName } : {}),
});
```

Переименовать `extractPackageScope()` → `extractPackageName()`, изменить возвращаемый формат: раньше `"packages/openapi"`, теперь `"openapi"`.

Обновить `vectorFilter` (line ~247):
```ts
const packageName = this.extractPackageName(query);
const vectorFilter = packageName ? { packageName } : undefined;
```

Тесты: добавить unit-тест `extractPackageName()` в `packages/rag/src/__tests__/`.

### Phase B — fix QdrantVectorStore filter

В `qdrant-vector-store.ts`, расширить `filter` тип и Qdrant query:
```ts
async search(
  vector: number[],
  k: number,
  filter?: { filePath?: string; packageName?: string },
): Promise<VectorSearchResult[]>
```
Must-clause: выбирать `packageName` если передан, иначе `filePath` (обратная совместимость).

Обновить `vector-store.ts` — добавить `packageName?: string` в тип filter (проигнорировать в теле).

### Phase C — migrate existing Qdrant collections

Скрипт `packages/rag/src/scripts/patch-qdrant-payload.ts`:
1. `GET /collections/{name}/points/scroll` с `with_payload: true`, по 100 points за раз
2. Для каждого point: если `payload.filePath` есть → вычислить `packageName` → `POST /collections/{name}/points/payload` (Qdrant set_payload, не upsert — точки не трогаем)
3. Идемпотентный: пропускать points у которых `packageName` уже есть

Запуск: `tsx packages/rag/src/scripts/patch-qdrant-payload.ts`

### Phase D — bench

- Задачи: trpc T1-T6 + hono H1-H6 при `VECTOR_BACKEND=qdrant`
- Проверить логи на наличие `packageName filter applied` (добавить `logger.debug` при срабатывании фильтра)
- Целевая метрика: ≥ 11/12 (= v1.65d baseline), желательно 12/12
- Run-file: `docs/benchmarks/runs/2026-05-26-v1.66-qdrant-scope-filter.md`

## 4. Acceptance criteria

- [ ] AC1 — `extractPackageName("packages/openapi/src/foo.ts")` → `"openapi"`, `extractPackageName("/abs/path/packages/server/utils.ts")` → `"server"`, `extractPackageName("src/utils.ts")` → `undefined`
- [ ] AC2 — При `VECTOR_BACKEND=qdrant` и trpc-задаче: лог содержит `packageName filter applied` хотя бы на 1 задаче из T1-T6
- [ ] AC3 — Patch-скрипт обновляет все points без `packageName` в payload (verify: scroll и проверить что поле есть)
- [ ] AC4 — Bench ≥ 11/12 при `VECTOR_BACKEND=qdrant`
- [ ] AC5 — `npm test` зелёный (610+ passing)

## 5. Risks & mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Patch-скрипт теряет данные (перезаписывает payload) | MED | Использовать Qdrant `set_payload` (merge, не replace); добавить dry-run флаг |
| `extractPackageName` не находит пакет в нестандартных монорепо | MED | Fallback: без фильтра (undefined), поведение идентично текущему |
| Scope filter слишком агрессивен — режет нужные символы из shared-пакетов | MED | Отключить фильтр для пакетов с именем `shared` / `utils` / `types`; логировать случаи срабатывания |
| Re-index нужен если patch не покрывает edge cases | LOW | После patch — запустить `/index` на проекте, убедиться что новые points имеют `packageName` |

## 6. Open questions

- Стоит ли добавить `packageName`-фильтрацию в BM25 в этой же итерации? (текущее мнение: нет — BM25 работает быстро, контекст-шум от него меньше)
- Нужно ли хранить `packageName` в CodeGraph/code-graph для v1.67? (скорее всего да — зарезервировать поле в `CodeSymbol` типе)

## 7. Alternatives considered

- **Prefix filter через Qdrant `match.text` (contains)** — отвергнуто: Qdrant text match — для full-text search, не для path prefix; exact `packageName` надёжнее
- **Хранить path-prefix как отдельное поле** — то же что `packageName` но многословнее, сложнее exact-match на стороне клиента

## 8. Out of scope

- BM25 scope filtering (v1.67+)
- `packageName` в CodeSymbol type (v1.67, когда добавляем SQLite symbol table)
- Qdrant cloud / sharding
