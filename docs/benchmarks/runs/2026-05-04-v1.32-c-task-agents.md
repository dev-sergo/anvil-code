## What worked

- ✅ **REFACTOR_SPEC fully delivers.** 3/3 byte-perfect on L3.1 — demonstrates the "AST may not fit; replace_in_file fallback OK" prompt suffix produces correct fallback behavior.
- ✅ **All healthy commits byte-perfect.** 6/8 commits are byte-identical canonical fixes — when the model succeeds, it does so cleanly. This validates the shared-prompt-core + spec-suffix architecture.
- ✅ **L4.1 wall-time win.** Healthy-run median dropped from 5:49 (v1.32-d) to 1:45 — ~3× faster. BUGFIX_SPEC's tighter `maxToolCalls=30` + `pruneHistory=true` materially help.
- ✅ **Bench harness assistant-driven end-to-end.** Reset → submit → SSE stream → capture diff → write run-file, all without operator intervention beyond llama-swap availability. `/tmp/bench-run.sh` (12 lines) is reusable.
- ✅ **Pathology guard inert across all 11 runs.** No same-fingerprint loops triggered — confirms guard is not over-eager.
- ✅ **L4.1 healthy fixes byte-perfect 3/3.** Same canonical diff each time (vs v1.32-d 1/3 byte-perfect). The model converges on the right fix when it finds it.

## What broke — done-after-structural-error pattern

The dominant failure mode across 3/11 runs (L1.1 #3, L4.1 #1, L4.1 #2): **Coder/Fixer encounters a structural-tool error on its first or second call (`add_route`, `replace_method`, `replace_function`), emits `done()` immediately afterward, no fallback to `replace_in_file`.** Pathology guard doesn't fire because the threshold (5 same-fingerprint errors) isn't reached — the model bails after 1 error.

Failure trace example (L1.1 #3):
```
[read_file] # src/server.ts (15 lines)
[add_route] error: no Fastify route calls (app.get/post/put/...) found in this file; cannot ...
[done] ok: changes finalized      ← 0 file changes
```

Why the model gives up so fast: SHARED_PROMPT_CORE includes "(d) call done() if you cannot make further progress" — that's escape language the model latches onto when the first try doesn't work. The structural-first hint amplifies it (model tries the structural tool, fails, concludes "this isn't tractable", calls done).

**Tractable mitigation — proposed v1.32-c.1:**
- Detect "Coder/Fixer about to call done() with 0 successful tool effects" and inject a strategy nudge: "you have 0 successful edits — try `replace_in_file` on the target file instead of structural tools, or read another file to confirm the location." Allow done() only after the nudge.
- ETA: ~½ day, +5-8 unit tests. Should lift L1.1 to 3/3 and L4.1 to ≥4/5.

## What broke — harness self-inflicted (resolved mid-session)

Runs #1-3 of L1.1 produced 0 commits due to a bench harness bug: `/tmp/bench-run.sh` used `git clean -fdx` between runs, which removed sandbox `node_modules` (gitignored). Sandbox `npm test` / `tsc` then fell back to the global `tsx` shim, which printed *"This is not the tsc command you are looking for"* and validation failed unconditionally.

Fix: `clean -fdx` → `clean -fd` (preserves ignored deps). One-time `npm install` in sandbox. After this, runs #4 onward (renumbered as L1.1 #1, #2, #3 above) executed under valid validation. The 3 invalid runs are excluded from the aggregate.

## Lessons / next iteration ideas

1. **The "done-after-error" failure mode generalizes.** It's not L1.1- or L4.1-specific — it's a SHARED_PROMPT_CORE issue (escape-language too prominent) interacting with structural-first specialization. v1.32-c.1 fix touches one shared loop hook, not three specs.
2. **Byte-perfect consistency is high once the model navigates correctly.** 6/8 commits byte-identical — when Coder finds the right file and tool, output is canonical. The variance moved from "edit shape" to "found vs not found".
3. **Wall-time win on L4.1 is large enough to be a v1.32-c selling point even with commit-rate ≤ v1.32-d.** Operator-experience improves regardless: failures fail fast (~100s) instead of spinning ~5 min.
4. **AC6 byte-identical regression criterion was the wrong AC.** Stochastic LLMs can't byte-match across runs even on the same task. Replace AC6 in future designs with "shape parity" (commits land, files touched ⊂ Planner's instruction set).
5. **Bench harness hardening.** `bench-run.sh` should `npm install` if `node_modules` missing, not assume. Sandbox setup checklist updated implicitly.
