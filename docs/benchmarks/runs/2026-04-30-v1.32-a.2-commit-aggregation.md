# Run 2026-04-30 — v1.32-a.2 Commit-landing aggregation fix

Direct response to v1.32-a.1's L4.1 outcome: Fixer applied a byte-perfect
fix to `user-service.ts`, validation passed, orchestrator logged "Committed
changes" — but `git status` showed the file modified-but-not-staged and
the auto-branch HEAD was unchanged from baseline. The commit log carried an
empty hash. v1.32-a.2 traces and fixes the bug: `runValidationLoop` was
mutating the on-disk file via SafeWriter but **never adding the Fixer's
paths to the outer `writtenFiles` array** that the commit step stages.

## What changed

### The bug, located precisely

In [`runTask`](../../../packages/agents/src/orchestrator.ts) (lines ~130-218):

```ts
const writtenFiles: string[] = [];                  // outer scope

// Coder phase: writes files, appends paths.
for (const change of dedupedFileChanges) {
  this.writer.execute(change);
  writtenFiles.push(change.path);
}

// Validation loop runs Fixer when needed.
const validation = await this.runValidationLoop(taskId, allFileChanges, mode, log);
//                                              ↑ writes Fixer files via SafeWriter
//                                              ↑ but DOES NOT TOUCH writtenFiles

// Commit phase stages writtenFiles only.
await this.git.commitChanges(taskId, msg, writtenFiles);
```

`runValidationLoop` (private method) was applying Fixer's edits via `this.writer.execute(fixed)` and updating its local `allFileChanges` array — but the outer scope's `writtenFiles` was never extended. So when the Fixer touched a path Coder didn't (typical bug-fix scenario: Coder edits routes/users.ts cosmetically, Fixer reads & fixes services/user-service.ts), the staging list passed to `commitChanges` was stale.

When `git.add(stalePaths)` was called, only the Coder's paths were staged. If Coder's paths had no actual diff vs the base commit (because Coder's "edit" was a no-op or rolled back by brace-balance), nothing was staged at all. `git.commit` against an empty stage produces no commit (simple-git returns an empty hash, not an error).

End-state: working tree dirty, auto-branch HEAD unchanged, "Committed changes" logged with `commitHash: ""`. The Fixer's correct fix sat in the working tree as a modified-but-uncommitted file.

### The fix

`runValidationLoop` now tracks written paths and returns them:

```ts
const fixerWritten = new Set<string>();
// ...
for (const fixed of dedupedFix) {
  try {
    this.writer.execute(fixed);
    fixerWritten.add(fixed.path);    // NEW
    // ... existing allFileChanges update ...
  }
}
// ...
return { passed, issuesCount, writtenFiles: [...fixerWritten] };
```

Caller in `runTask` merges before commit:

```ts
const validation = await this.runValidationLoop(...);
for (const p of validation.writtenFiles) {
  if (!writtenFiles.includes(p)) writtenFiles.push(p);
}
// commitChanges now sees Coder ∪ Fixer paths.
```

Dedupe is by inclusion-check rather than `Set` to preserve insertion order (Coder's paths first, then Fixer's additions) — small UX nicety for commit messages and event payloads downstream.

### Tests (+2 new, 12 total in `orchestrator.test.ts`)

- **`commits Fixer-produced paths even when Coder did not touch them`** — Coder writes `src/routes/users.ts`, TypeChecker fails on first call, Fixer writes `src/services/user-service.ts`, TypeChecker passes second call. Assertion: `git.commitChanges` called with both paths in its `files` arg.
- **`dedupes the staged file list when Coder and Fixer touched the same path`** — Coder writes `src/foo.ts`, Fixer also writes `src/foo.ts` after typecheck retry. Assertion: staging list is exactly `['src/foo.ts']`, no duplicate.

These tests directly cover the fix at the unit boundary — no LLM dependency, fast feedback, repeatable. They pin the bug shut: any future refactor of `runValidationLoop` that drops the `writtenFiles` return must break them.

### What's the upstream problem

Across the four L4.1 runs since v1.32-a:
- v1.32-a: Fixer navigated to user-service.ts (read), proposed correct fix (write rejected by old policy).
- v1.32-a.1: Fixer navigated, applied correct fix, validation passed, commit didn't land (bug v1.32-a.2 fixes).
- **v1.32-a.2 #1: Fixer made 0 tool calls in both attempts. Bailed.**
- **v1.32-a.2 #2: Fixer made 0 tool calls in both attempts. Bailed.**

**2 navigation attempts, 2 bails — Fixer is non-deterministic on this task with a ~50% bail rate.** When Fixer bails (returns text without tool calls), the validation loop terminates without producing any fix, and v1.32-a.2's commit-aggregation path is never reached — the loop returns `passed: false, writtenFiles: []`.

**Cause hypothesis:** the Fixer's SYSTEM_PROMPT has grown substantially across v1.32-a + v1.32-a.1 (test-failure-points-to-production guidance + read-grants-write rule + scope-discipline retread). The combined prompt is now ~25 lines longer than v1.30.5's. The model may parse this as a complex set of constraints, see an "Allowed write targets" list with only `routes/users.ts`, conclude it can't help, and bail with text only.

This isn't v1.32-a.2's bug — it's a Fixer-prompt-clarity issue surfacing because the L4.1 task is hard. v1.32-a.2 ships its specific fix with its specific evidence (unit tests). The Fixer-bails issue is the next iteration's target.

## What worked

- ✅ **Bug located precisely.** `runValidationLoop` writing files via SafeWriter but not propagating paths to the outer staging list — identified in code, fixed in code, covered by tests.
- ✅ **Unit tests pin the bug shut.** Two tests directly exercise the aggregation: one for the Coder/Fixer-different-path case (the live failure mode), one for the same-path-dedupe case (defensive against future regressions).
- ✅ **Insertion-order preserved.** Staging list is `[...coderPaths, ...newFixerPaths]` — predictable for downstream consumers (commit messages, event data, telemetry).
- ✅ **No regression on existing tests.** 382/382 green; the 10 prior orchestrator tests still pass with the new `writtenFiles` return shape.
- ✅ **Fix is small and reversible.** ~10 lines of orchestrator code + 2 unit tests. If a future redesign of validation flow makes this aggregation differently-shaped, easy to lift.

## What broke

- ⚠ **L4.1 live bench inconclusive.** Two consecutive runs had Fixer bail with 0 tool calls, so the commit-aggregation path was never exercised. The fix is ONLY validated at the unit level. To call this "L4.1 closes end-to-end" we'd need a live run where Fixer navigates → fix lands → commit aggregates → auto-branch HEAD has the new commit. v1.32-a.1's lone navigation-success was the closest we've gotten.
- ⚠ **Fixer non-determinism is now the dominant failure mode** on navigational bug-fix tasks. ~50% bail rate over 4 runs. This is the natural successor problem for v1.32-a.3.
- ⚠ **Prompt accretion across v1.32-a + v1.32-a.1 + v1.32-a.2** has produced a Fixer prompt notably more complex than v1.30.5's. The model may benefit from a consolidation pass (same rules, fewer words).

## Lessons / next iteration ideas

1. **Unit testing pinned a real bug.** The aggregation fix is correct in isolation, with mocked validators and writers. When the live bench can't reach the path, the unit tests are the rigorous backstop. Engineer's quality bias paid off: the fix doesn't depend on a live demonstration to be trustworthy.
2. **Fixer-bails is the dominant remaining failure on L4.1.** v1.32-a.3 should focus there. Two angles:
   - **Prompt consolidation** — collapse the v1.32-a/a.1 additions into clearer language. Possibly move the "read-grants-write" rule to the user message rather than system prompt, where it's adjacent to the "Allowed write targets" list it expands.
   - **Loop-level signal** — if the Fixer returns 0 tool calls on two consecutive attempts, retry with a stronger system reminder ("you have not produced any edits — the validation issues require you to read and modify production code; navigate now"). Currently the loop bails after one no-tool-calls round.
3. **Orchestrator's `commitChanges` swallows empty commits silently.** The commit step received `writtenFiles=['routes/users.ts']` (Coder's path), called `git.add(['routes/users.ts'])`, then `git.commit` with no actual diff to stage. simple-git returns an empty hash; orchestrator logs "Committed changes" anyway. Worth adding a guard: if `commitResult.commit` is empty, log a warning and emit a `commit_empty` event rather than `commit`. Out of scope for v1.32-a.2; possible v1.32-a.4.
4. **The four-run series is itself useful evidence.** Same task, four runs, four different outcomes — each reveals a distinct part of the system. v1.32-a.2 closes the most subtle one (commit semantics) without addressing the more visible Fixer-navigation noise.

## Decision

**v1.32-a.2 lands** — the bug is real, the fix is correct, the unit tests pin it. Live validation deferred until Fixer navigation is more reliable.

**Recommended next step — v1.32-a.3** (Fixer reliability, ~2-4h):
- Investigate Fixer prompt complexity. Consider moving dynamic policy rules from system prompt to user message (where allowed-paths list lives).
- Add a no-tool-calls retry: if Fixer returns text with no tool_calls in attempt N, the loop emits a stronger nudge in N+1 before bailing.
- Re-run L4.1 — expect Fixer to navigate reliably, then v1.32-a.2's aggregation fix will get its end-to-end demonstration.

After v1.32-a.3, a single L4.1 success would close the v1.31.x → v1.32-a.x arc: bug-inject baseline → byte-perfect fix → committed auto-branch → operator-grade green commit. That's the defining milestone for "operator-grade bug-fix workflow."
