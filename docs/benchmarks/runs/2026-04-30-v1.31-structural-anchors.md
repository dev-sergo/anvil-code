# Run 2026-04-30 — v1.31 Structural anchor edits

The architectural shift the v1.30.5 retrospective recommended: replace
line-coordinate edits with **AST-anchored structural tools**. The Coder no
longer thinks in line numbers for source code — it names a class, method,
function, route, or import, and the runtime resolves the AST to a line
range. Eliminates whole categories of placement bugs (consumed brace,
duplicate context, off-by-one, method-outside-class) by construction.

## What v1.31 changed

[packages/agents/src/structural-edits.ts](../../../packages/agents/src/structural-edits.ts) — new module, ~530 lines:

- `locateAddMethod(content, container, source)` — walks `ts.SourceFile`, finds the named class, parses `source` as a single method declaration, computes the closing-brace line, returns `{ kind: 'insert', line, text }` with re-indented body.
- `locateReplaceMethod(content, container, name, source)` — finds the method, returns line range + new body. Cross-checks `source`'s declared name matches `name` to prevent silent renames.
- `locateReplaceFunction(content, name, source)` — same shape, top-level FunctionDeclaration.
- `locateAddRoute(content, http_method, route_path, body, params?)` — Fastify-aware. Walks for `app.METHOD(path, ...)` calls, copies the actual instance name (`app` / `server` / `fastify` / `instance` / `route`), copies the indent style (2 / 4 / tabs), inserts a new route after the last existing one. Errors on duplicate (method+path) or unknown HTTP method or unsafe path characters.
- `locateAddImport(content, source, names?, defaultName?, typeOnly?)` — idempotent: returns `noop` when everything's already there, merges names into existing import from same source, or inserts a fresh `import` after the last existing import (or top of file).
- `locateAddExport(content, source)` — appends after last top-level export, else after last import, else top.

[packages/agents/src/working-set.ts](../../../packages/agents/src/working-set.ts):

- `insertBefore(relPath, line, text)` — squeeze content before line N. Trailing `\n` in text treated as line terminator (not extra blank line). Used by all `kind: 'insert'` structural edits.

[packages/agents/src/tool-calling-coder.ts](../../../packages/agents/src/tool-calling-coder.ts):

- 6 new entries in `TOOL_DEFINITIONS`. 6 new dispatcher cases routing through `executeStructuralEdit(toolLabel, filePath, ws, policy, locator)` — same scope discipline (v1.30.1) and brace-balance defense (v1.30.5) as `replace_in_file`.
- `SYSTEM_PROMPT` rewritten: structural tools listed first as PREFERRED for TS/JS edits, `replace_in_file` demoted to "fallback for non-source content (markdown/JSON/YAML/plain text) or unusual code shapes". Added explicit warning: line numbers shift after every edit; re-read before any subsequent line-coord call.
- `WRITE_EMITTING_TOOLS` + `FILE_ARG_KEY` — extended `coder_file_ready` event emission to all structural tools; uses `file` arg for structural tools and `path` for line-coord tools.

[packages/agents/src/tool-calling-fixer.ts](../../../packages/agents/src/tool-calling-fixer.ts):

- `FIXER_SYSTEM_PROMPT` updated symmetrically — structural tools as PREFERRED, with explicit guidance: `add_import` requires the `names` array (an empty `names` produces a useless side-effect `import 'src';` that doesn't bring any symbol into scope).

[packages/model-router/src/types.ts](../../../packages/model-router/src/types.ts):

- `ToolParamSchema` extracted to support `items` for array-typed parameters (`add_import.names`).

**Tests:** 5 new `WorkingSet.insertBefore` tests; 45 `structural-edits` unit tests covering each helper (happy path, container/method missing, duplicate, type-only mismatch, namespace-import refused, single-line-class refused, etc.); 12 dispatcher integration tests (one per tool happy path + scope discipline + forbidden-config + locator-error surfacing). 356/356 tests green; +62 vs v1.30.5 baseline (294).

## Sandbox L2.1 — Cross-file middleware

```
Add a request-logging middleware in src/middleware/request-log.ts that logs
{method, url, statusCode, durationMs} on every response. Register it in
src/server.ts via app.addHook("onResponse", ...).
```

| | |
|---|---|
| Plan size | 2 steps |
| Files touched | `src/middleware/request-log.ts` (new), `src/server.ts` |
| Validation | typecheck pass, test pass |
| Commit | **completed + committed** |
| Wall time | ~10m |

**Tool sequence (clean for create + add_import):**
```
create_file(src/middleware/request-log.ts, <plugin source>)
read_file(src/server.ts)
add_import(src/server.ts, ./middleware/request-log.js, [requestLogPlugin])
replace_in_file(src/server.ts, 6, 6, "await app.register(usersRoutes);\napp.register(requestLogPlugin);")
done()
```

**Diff:**
```diff
+ import { requestLogPlugin } from './middleware/request-log.js';
 
 const app = Fastify({ logger: true });
-
+await app.register(usersRoutes);
+app.register(requestLogPlugin);
 await app.register(usersRoutes);     ← duplicate (line-shift bug)
```

**Issues / observations:**
- **`add_import` worked perfectly** — atomic, idempotent, sorted output.
- **`replace_in_file` line-shift bug.** After `add_import` shifted the line numbers of `src/server.ts` (inserting an import at the top), the model's `replace_in_file(start_line=6, end_line=6, ...)` used STALE coordinates (from the original `read_file`). What was line 6 in the read became line 7 after `add_import`. Result: model replaced the blank line at new-line-6 (formerly line 5's blank), and the original `await app.register(usersRoutes);` at new-line-7 was preserved as-is — so the file now registers `usersRoutes` twice.
- Reviewer + validation didn't catch the duplicate (the file compiles; tests pass; double-registration is a runtime quirk in Fastify, not a typecheck error).
- This is exactly the failure mode v1.31 was designed to AVOID via structural tools — but `app.register(...)` (plugin registration, not a route) doesn't fit `add_route`'s contract, so the model fell back to `replace_in_file`. Combined with no re-read after the prior structural mutation, line coords drifted.

**Mitigation landed mid-run:** SYSTEM_PROMPT updated to warn explicitly about line-shift after structural edits and require re-read. Will validate in a subsequent run.

## What worked

- **`add_route` on /version: byte-perfect.** The same edit v1.30.4 had to wrap with brace-balance defense, v1.30.5 had to wrap with rollback, just *works* — the AST resolves the right insertion line, the body re-indents to match the file's style, and the surrounding code is preserved by construction. Three tool calls (read, add, done) replace ~25 line-coord rounds. Time dropped from 32m to 12m.
- **`add_method` on getSize: structurally correct.** The "method outside class" failure mode that plagued v1.30 is gone — `add_method` finds the class's closing brace and inserts before it. No way to land outside the class by construction.
- **`add_import` on L2.1 (Coder) and Fixer phase: idempotent, properly merging.** Even when used incorrectly (empty names) it doesn't corrupt files — it just produces a useless side-effect import.
- **Wall times dropped substantially.** /version: 32m → 12m (~3× faster). getSize: ~7m. Tool calls per task collapsed from 25-50 to 3-7.
- **scope discipline + brace-balance check carry over cleanly.** The structural dispatcher reuses `executeStructuralEdit(toolLabel, filePath, ws, policy, locator)` which applies the same v1.30.1 + v1.30.5 defenses to every new tool. No regression on those properties.
- **45 unit-tests for structural-edits + 12 dispatcher integration tests** verify each tool's contract independently of LLM-generated input. 356/356 green.

## What broke

- **Sandbox L1.1 file-anchor non-determinism.** When the task says "add a route" without naming a file, Coder picks the wrong file half the time (`server.ts` over `routes/users.ts`). Past iterations were deterministic 10/10 on this task — v1.31 introduces variability because the new prompt steers more aggressively toward structural tools, and `add_route` errors on `server.ts` (no existing routes), at which point the fallback path is brittle.
- **L2.1 line-shift bug.** After `add_import` mutated `server.ts`, the model called `replace_in_file` with stale coordinates (from its earlier `read_file`). Resulted in `usersRoutes` being registered twice. Validation didn't catch it; the bad commit landed. Fix: SYSTEM_PROMPT now requires re-read after every mutation. To validate next run.
- **Fixer's `add_import(file, source)` without names** — model uses the new tool but omits the `names` array, getting a useless side-effect import that doesn't fix the typecheck errors. Pattern was deterministic across both /version and getSize Fixer phases (8 such calls total). Fix: Fixer SYSTEM_PROMPT now explicitly warns. To validate next run.
- **rag-system-target validation passes are still flaky** because the project has pre-existing typecheck/test errors unrelated to the Coder's atomic edit. The Fixer takes the bait and tries to "fix" them, which is the wrong scope. Not a v1.31 regression — pre-existed v1.30.x.

## Lessons / next iteration ideas

1. **The architectural shift is decisively the right call.** Six iterations (v1.30 → v1.30.5) of whack-a-mole on Coder line-coord failure modes is replaced by ONE iteration of structural tooling, and the Coder layer is solved for atomic file-anchored tasks. The pattern matches the prediction: each line-coord failure was at the AST-anchor layer (off-by-one, consumed brace, duplicate context); removing line coords from the model's hands removes those failure modes by construction.
2. **Variability moved from "edit correctness" to "file/argument selection".** v1.30.5's failures were "Coder edit corrupts a file." v1.31's failures are "Coder picks the wrong file" or "Fixer omits the `names` arg." These are *meta-decisions* about which tool to invoke and how — fundamentally different and arguably more tractable. They yield to prompt iteration; the previous failures didn't.
3. **Two prompt fixes landed mid-run** (line-shift warning, Fixer add_import names guidance). Both are direct responses to observed bench evidence. To be validated in the next bench cycle.
4. **`add_route` bootstrap is a real gap.** Files with zero existing routes can't be anchored. For v1.31 the workaround is "use `replace_in_file`"; for v1.32 a clean answer might be `add_first_route(file, build_function_name, ...)` that finds a `Fastify()` instantiation and inserts after it — but this is rare enough to defer.
5. **Validation on rag-system-target is noisy.** Pre-existing unrelated errors mean Coder's clean edit can still get `commit_skipped`. The auto-branch is mergeable; an operator gate (or a more conservative Fixer scope policy) could land it. **v1.32 candidate**: Fixer should refuse to edit files outside the Coder's scope unless the typecheck error explicitly references those files.
6. **`getSize()` placement was deterministically broken in v1.30 and is deterministically correct in v1.31.** This is the cleanest demonstration of the architectural win — the same task, same model, same prompt structure, but a different *primitive* (line-coord vs symbol-anchored) flips the outcome from broken to perfect.

## Decision

**v1.31 lands in tree.** The architectural shift to structural tools is correct, the Coder layer for file-anchored atomic tasks is solved, and the new failure modes (file-selection variance, Fixer add_import-without-names) are smaller-class problems with prompt-level fixes already drafted.

**Next iteration recommendation — v1.31.1 (small, ~2-3h):**
- Re-run the bench with the two SYSTEM_PROMPT fixes already in tree (line-shift re-read + Fixer add_import names guidance).
- Goal: confirm L2.1 commits without the duplicate-register bug; confirm rag-system-target /version + getSize land in commit-able state (Fixer doesn't shoot the test files).
- Optional: stricter Fixer scope policy (don't write to files outside Coder's scope unless an issue explicitly references them).

After v1.31.1, the natural next step is **v1.32 sub-agents** — each agent (BugFixAgent, RefactorAgent, FeatureAgent, MigrationAgent) gets a small specialty toolset built on the v1.31 structural primitives. The path forward from "structural tools work" to "specialized agents pick the right tool" is now incremental.
