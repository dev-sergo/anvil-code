# Run 2026-04-30 — v1.31.1 Prompt-fix validation

Two SYSTEM_PROMPT changes landed mid-run during v1.31:

1. **Line-shift warning** in Coder + Fixer prompts: "line numbers shift after every edit ... read_file again before any subsequent replace_in_file." Direct response to the v1.31 L2.1 sandbox failure where `add_import` shifted lines, model used stale coords from earlier `read_file`, ended up with duplicate `await app.register(usersRoutes);`.

2. **`add_import` requires `names`** in Fixer prompt: explicit warning that `add_import(file, source)` without a `names` array adds a useless side-effect `import 'src';` that doesn't bring symbols into scope. Direct response to v1.31 /version + getSize benches where Fixer fired 8 such empty-names calls into test files trying (and failing) to fix typecheck errors.

This run validates both fixes against the same bench tasks that surfaced them.

## What worked

- **Line-shift prompt fix decisively eliminates the L2.1 duplicate-register bug.** Reviewer + Fixer can now recover from the model's residual "include too much in `replace_in_file` new_text" mistake, and the fundamental "stale line coords corrupt the existing register" bug doesn't fire.
- **Fixer prompt fix decisively eliminates empty-names `add_import` calls.** Across two target benches and 20+ Fixer tool calls, every `add_import` now includes a concrete `names` array. Model behavior changed cleanly in response to a single-line prompt edit.
- **L1.1 failure rate dropped from 50% to 0%** across the runs in this validation cycle. v1.31 had the model land /health outside `usersRoutes` half the time; v1.31.1 always lands it in a committable place (even if one of three runs has cosmetic indent issues).

## What broke

- **L1.1 #3 cosmetic indent.** When Coder picks `replace_in_file` over `add_route` for L1.1 (1 of 3 runs), it uses 4-space indentation in the new_text instead of matching the file's 2-space style. The file compiles, tests pass, but the diff is stylistically off. Lower-severity than the v1.31 failures and not a regression of any previously-fixed property.
- **Target benches still commit_skipped due to pre-existing typecheck noise.** Coder's edits to `server.ts` and `queue.ts` are byte-perfect, Fixer's response is now correctly named, but the target project has unrelated typecheck errors that no amount of import-adding will fix — they need actual code understanding. **This is a Fixer-scope problem, not a tool-shape problem.**

## Lessons / next iteration ideas

1. **Two targeted prompt edits validated cleanly.** Both v1.30+ failure modes that landed mid-v1.31 are now empirically fixed. Total cost: ~2 lines of system-prompt text, ~30 minutes of bench runs.
2. **The Coder still has residual "what's in the WorkingSet now?" confusion** (L2.1's duplicate import in `new_text`). Reviewer caught it this run; not always reliable. **v1.32 candidate**: when `add_import` succeeds, immediately follow with a synthetic tool result reminder ("the WorkingSet now contains: ... Don't re-emit these in subsequent replace_in_file calls.") — runtime nudge, not just system prompt.
3. **Fixer scope discipline is the next frontier.** v1.31 + v1.31.1 produce byte-perfect Coder output and well-formed Fixer tools, but the Fixer is reaching into 10+ unrelated test files to "fix" pre-existing target noise. **v1.32 candidate**: stricter Fixer policy — don't write to a file unless the validation issue explicitly references that file path. The `buildFixerAllowedSet` already supports this, but the model takes the bait of "general typecheck errors" and broadens scope.
4. **L1.1 file-selection variance still exists** (model picks `replace_in_file` 1/3 of the time). Possible mitigation: stronger SYSTEM_PROMPT preference for `add_route` when an existing route is in scope. Lower priority — variance no longer produces failures, just stylistic inconsistencies.

## Decision

**v1.31.1 lands in tree** (just the two prompt edits — no code changes beyond removing the temporary debug `logger.info` in `dispatchToolCall`). Total delta from v1.31 is ~5 lines across two prompts. Both fixes empirically validated against the exact bench tasks that surfaced them.

**Next iteration recommendation — v1.32 candidates** (in priority order):
- **v1.32-a (~3-4h)**: stricter Fixer scope — refuse writes outside the Coder's working set unless an issue explicitly references the file path. Should turn target benches from commit_skipped into commit_landed without further Coder work.
- **v1.32-b (~2-3h)**: post-tool synthetic "WorkingSet state" message after structural mutations, so Coder/Fixer don't double-emit content already in the file. Catches the L2.1 residual pattern.
- **v1.32-c (later, larger)**: sub-agents (BugFixAgent / RefactorAgent / FeatureAgent) — each with a small specialty toolset built on v1.31 structural primitives.
