## What worked

- **L2.1 lifted from variance hell to perfect deterministic 10/10.** The Example-A pattern is reproduced byte-for-byte in both runs. This is the strongest evidence yet that local models follow worked examples better than rules — the same model with the same task that picked `onRequest` twice on v1.25 now picks `onResponse` twice on v1.26 with identical output structure. Few-shot examples are the right tool for "model knows the rule, model still ignores the rule" failures.
- **No regression on L1.1.** Both runs land cleanly, atomic-task semantics preserved.
- **L2.3 cumulative variance reduced.** Three iterations in (v1.24 → v1.25 → v1.26) the median has lifted: best-of stays 9-class, no hard infra-fails this run, mean lifted from ~4.6 (with fail) to 7.9.
- **Wall time also improved on L2.1** (~2m vs v1.25's 3m+). Slightly less Fixer churn because the initial Coder output is closer to spec.

## What broke

- **L2.3 #1 — partial commit (2/3 files).** Planner emitted 3 steps; step3 (DELETE endpoint) hit Reviewer rejection 3 times. Per-step recovery let the other two steps commit; the working tree on the auto-branch contains the partial state, but the user has no explicit "incomplete" signal — the SSE `done` event still fires. This is the v1.28 target (silent partial events).
- **Reviewer-rejection-cycle on cumulative L2.3 #1.** Whatever the Coder produced for step3, the Reviewer kept saying "issues found" and the Fixer didn't converge in 3 attempts. Without seeing the Reviewer's complaints in the run-file workflow, hard to attribute exactly — but it's a sign that Reviewer can be the variance source even when Coder/Fixer prompts are tight.
- **Plan-size variance is still the main cumulative-task multiplier.** v1.26 ran #1 as 3 steps (partial) and #2 as 1 step (full GREEN). Same prompt rules to Planner, very different output. v1.27 (per-agent context tailoring) and a Planner-side few-shot for rule 6a are the next levers.

## Lessons / next iteration ideas

1. **Few-shot examples are demonstrably the most effective single intervention for systematic semantic misses.** L2.1 N=2 went from mean 6.4 to mean 10.0 with zero code logic changes — only a prompt edit. Apply the same pattern to other agents that show systematic misses (Planner is next).
2. **v1.27 (per-agent context tailoring) should include a Planner few-shot** for rule 6a (cross-file coupled = single step). The variance on L2.3 cumulative between 1-step and 3-step plans is the dominant remaining source of error, and Planner is the right place to attack it.
3. **v1.28 (silent partial completion events) is now overdue.** L2.3 #1 here is exactly the use case: the task ran "successfully" (Status: completed, validation passed, commit landed), but a user reading only the `done` event would not know that 1/3 of the files asked-for never made it. Adding a `commit_partial` event with `incomplete_files: string[]` and `failedSteps: string[]` would close the gap.
4. **Reviewer is the new variance source on cumulative state.** Worth investigating: does the Reviewer have an analogous failure-mode (e.g. systematically rejecting valid Coder output for surface reasons)? If yes, few-shot examples for Reviewer too. Optional v1.26.1 if the pattern holds in another run.

## Decision

Land v1.26 in tree. The L2.1 result is decisive: the failure mode that sat unaddressed across v1.24 (commit_skipped) and v1.25 (variance band 5.4–7.4) collapsed to mean-10 deterministic on a single prompt edit. L2.3 cumulative still has variance but the floor (no infra-fails, no commit_skipped) is the highest seen across the three Phase 2 iterations.

Next: **v1.28 (silent partial events)** — short, attacks the L2.3 #1 UX gap directly. Then **v1.27 (per-agent context tailoring)** with Planner few-shot to attack the remaining cumulative-task variance.
