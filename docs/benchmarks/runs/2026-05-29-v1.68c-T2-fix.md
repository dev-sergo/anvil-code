## Infrastructure bugs found (all causing "fetch failed")

### 1. LLM_URL not loaded on restart

**Root cause:** Server restarted via `npm exec tsx packages/api/src/index.ts` without loading
`.env`. No `dotenv` in the app — it reads `process.env` directly. The env var `LLM_URL` was not
set, so `config.llamacpp.url` defaulted to `http://localhost:8080` (nothing there).

**Evidence:** API server startup log:
```
WARN: LLM backend not available — tasks will fail until it is running
  url: "http://localhost:8080"
```
(vs `.env`: `LLM_URL=http://172.20.10.4:8080`)

**Fix:** Restart with `node --env-file=.env packages/api/dist/index.js`. Confirmed by
`/health → backendUp: true`.

### 2. Fixer deleting Coder-produced test files

**Root cause:** `interceptToolCall` in `BUGFIX_SPEC` only blocked `create_file` on test paths;
`delete_file` was allowed. Fixer deleted `requestTimeout.test.ts` because it couldn't fix the
test, which caused `git add <file>` to fail at commit time.

**Fix:** Extended `interceptToolCall` to also block `delete_file` on test paths.

### 3. Fixer prompt: "find production code" for Coder-produced test bugs

**Root cause:** `FIXER_SYSTEM_PROMPT` SCOPE section said "Test files are not writable unless the
Coder produced them — find the production code instead." The "find production code" instruction
was ambiguous and made Fixer ignore test-setup bugs even in Coder-produced files.

**Fix:** Reworded SCOPE to say: "If the Coder produced the test file AND the issue is a
test-setup bug, fix the test file directly — do NOT chase production code for test-setup
problems."
