# Run 2026-04-29 — v1.30.4 Coder cargo-cult prompt fix

Surgical prompt change. v1.30.1 + v1.30.3.1 benchmarks repeatedly produced a
specific failure: Coder asked to "add a /version endpoint that returns
{ version: '1.0.0' }" would read the existing /health route, then write a
/version handler whose body was a clone of /health's body — same `OllamaClient`
healthcheck, same `{ status, ollama, uptime }` return — instead of the
requested literal. Validation would then fail (typecheck/test), Fixer would
run, Fixer couldn't recover the actual issue from validation output, and the
task would either commit_skipped or hit the Ollama-crash ceiling.

v1.30.4 is a 30-minute prompt fix targeting this specific failure mode.

## What v1.30.4 changed

[packages/agents/src/tool-calling-coder.ts](../../../packages/agents/src/tool-calling-coder.ts), in the Coder's `SYSTEM_PROMPT`, after the workflow steps, a new section:

```
CONTENT COMES FROM THE TASK DESCRIPTION — NOT FROM SIBLING CODE. This is the
most common silent failure mode:
- read_file is for understanding STRUCTURE (where to put the new code, what
  indentation/imports/patterns the file uses) — NOT for copying logic. The new
  code's BEHAVIOR is specified in the task description.
- If the task says `add a /version endpoint that returns { version: '1.0.0' }`,
  your new_text MUST contain `return { version: '1.0.0' }`. It MUST NOT contain
  a clone of the /health handler's body just because /health was the nearest
  example you read.
- Read sibling routes/methods to learn HOW the file is wired (handler
  signature, registration style, helper imports). Then write the code the TASK
  asked for, with that wiring around it.
- A handler that echoes its neighbour's body instead of doing what was asked is
  wrong even if the file compiles. Validation will not necessarily catch it;
  the operator will.
```

No code logic changed. 280/280 unit tests still pass.

## What worked

- **Content-from-task prompt rule:** observable, specific behavioural change. The model stopped cloning sibling-route bodies.
- **No regression elsewhere:** scope discipline still holds (only `packages/api/src/server.ts` modified, no collateral). v1.30.1 + v1.30.3.1 stack carries through.
- 280/280 unit-tests still green; 12/12 packages build.

## What broke

- **Coordinate-off-by-one in `replace_in_file`.** Model picked `(start_line, end_line)` that included the closing `});` of /health, then wrote replacement text that didn't include a closing for /health. The closing brace was consumed by the replace and never restored. Result: file is syntactically broken even though the new code's content is right.
- **Fixer still crashes Ollama on broken files.** This is the v1.30.3.1 ceiling — when Fixer tries to fix a structurally-broken file, the conversation grows long, llama runner OOMs.

## Lessons / next iteration ideas

1. **The prompt-engineering layer is now exhausted on Coder content.** v1.30.4 was the last "cheap prompt fix" with predictable yield. Remaining failures are structural (placement, brace balance) and need code-level intervention, not more rules.
2. **v1.30.5 — verify-syntax tool after `replace_in_file`** (~3-4 hours) is now the highest-priority structural fix. After every replace_in_file, run a fast syntactic sanity check on the affected file's content (brace/paren/bracket balance, trivial JS-tokenizer pass). If unbalanced, the dispatcher returns `error: edit at lines X-Y left the file unbalanced (expected matching '})', got '});'); please retry with adjusted line range or include the missing closing.` Model gets a tool message and can adjust within its loop, before validation ever runs.
3. **Reviewer rejects empty Coder output (v1.30.2)** still pending; cheap, useful, independent.
4. **Smaller LARGE model for Fixer might still help** if Fixer fires on structural issues — those tend to require many round-trips. But after v1.30.5 reduces the rate at which Fixer fires (because syntax errors are caught upstream), the Ollama-crash ceiling becomes much rarer.

## Decision

**v1.30.4 lands in tree.** It produces a measurable, specific behavioural change (content correctness on /version for the first time) for a 30-minute prompt edit. No regressions.

**Next: v1.30.5 verify-syntax tool.** That's the structural-correctness equivalent of what v1.30.4 is for content. Combined, they should produce the first end-to-end GREEN commit on rag-system-target — content correct + placement correct + Fixer rarely needed.
