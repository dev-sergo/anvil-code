# Run 2026-04-29 — v1.29 Scale validation on rag-system codebase

First scale benchmark. Phase 2 closed with deterministic 10/10 on a 5-file
sandbox. The user's stated goal is 100-1000 file projects. This run measures
what actually happens at the next scale step: **the rag-system project itself
(91 TS files, 65 with symbols, 6717 LOC, 12 packages)** — used as a target via
an isolated copy at `~/rag-system-target`.

**Headline result:** the system as built does not survive 91 files. Failure
modes that never appeared on sandbox emerged immediately. Phase 3 (tool-calling
Coder) is now empirically justified, not just theoretically.

## What we measured

### Repo-map composition is wrong on this scale

Default `maxBytes: 6000` was sized for sandbox (5 files). At 91 files / 65 symbol-bearing files, the budget overflows after only 26 files and the rest are footer-truncated. Worse, the rendered top-26 list has alphabetic-sort bias: test files (`packages/agents/src/__tests__/...`) win over production code because `__tests__` sorts adjacent to file paths in `agents`. Production files like `packages/job-system/src/queue.ts`, `packages/orchestrator.ts` etc. competed for budget on equal footing with `coder-streaming.test.ts` and `partial-json.test.ts`.

For the model, this means: the repo-map shown to it advertises lots of test infrastructure and only some production code, with a footer saying "39 more files omitted." The model's "what exists in this project" view is materially incomplete and biased toward the wrong subset.

### Atomic Task #1 — `GET /version` in `packages/api/src/server.ts`

**Two attempts.**

**Run #1: hard infra fail (transient).**
- Plan generated (1 step, ~28s).
- Step started, then 30s into the Architect/Coder LLM call: `Ollama /api/chat 500: {"error":"llama runner process has terminated: %!w(<nil>)"}`. Llama runner subprocess crashed. Task marked `failed`. Wall: 21s before failure.
- Possibly an OOM from the long prompt, possibly a transient Ollama issue — second attempt cleared it.

**Run #2: nothing landed.**
- Plan generated (1 step, ~30s).
- Step approved by Reviewer at +2:38 — the Coder output looked plausible to the Reviewer.
- **Initial write failed immediately**: Coder's `search` block in the `modify` action did not match any substring of `packages/api/src/server.ts`. Tolerant fallback (v1.24) did not engage either — failure was structural, not whitespace.
- `retryFailedEdits` (v1.23 retry-with-real-content) ran: Fixer was given the literal current file content as `<<<<<<< CURRENT FILE`. Even with that, Fixer's regenerated edit ALSO didn't match — `Retry also failed; file remains unchanged`.
- TestRunner OK, TypeChecker FAIL (no `/version` route present, no behaviour added).
- Validation loop ran 3 Fixer rounds: every single one's `writer.execute` threw "search not found" — caught by v1.25.1's try/catch (`'Validation-Fixer write failed; treating as another validation issue and continuing'`). Three rounds, zero successful patches.
- Final state: working tree clean. `commit_skipped` correctly fired. Wall: 6m 13s. **Score: 0/10.**

**What this tells us:** even a one-file atomic task — the kind that landed deterministic 10/10 on sandbox — fails on a 91-file project because the Coder's `search` block doesn't survive the longer/noisier context. Five separate LLM calls (Coder + 1 retry-Fixer + 3 validation-Fixer rounds) all independently produced search blocks that didn't byte-for-byte match `server.ts`.

### Atomic Task #2 — `getSize()` method in `packages/job-system/src/queue.ts`

**Run #1.**
- Plan generated (1 step).
- Step approved by Reviewer at +2:51.
- **Initial write failed** — same pattern: Coder's first edit's `search` block didn't match `queue.ts`.
- Retry-with-real-content **succeeded** at +3:56 (Fixer with literal current content patched the file).
- TestRunner OK, TypeChecker FAIL — typecheck found problems somewhere (the model probably forgot to update something else, or modified the test file in an inconsistent way).
- 2 validation Fixer rounds. Several `Validation-Fixer write failed` messages (v1.25.1 catches), some `File patched` / `File written` messages — at this scale Fixer is hit-and-miss.
- Validation never passed → `commit_skipped`. Wall: 7m 49s.
- **Working tree at end of run**: `packages/job-system/src/queue.ts` modified (target, OK), **`packages/agents/src/__tests__/orchestrator.test.ts` modified (scope creep, unrelated test file)**, **`packages/agents/src/__tests__/vitest-setup.ts` newly created (scope creep, unrelated file)**. None committed.
- **Score: ~4/10** — main file modified, but scope creep into unrelated test infrastructure is a fresh failure mode, and validation never passed.

**Why scope creep into test files?** The repo-map is dominated by test-file symbols (alphabetic sort puts `__tests__` adjacent to other `agents` files). When the model navigates the project, the most prominent "what exists" view is biased toward test infrastructure. The model decided the right way to add `getSize()` was to also modify a test setup. This is a direct consequence of the budget-truncated, alphabetic-sorted repo-map.

## Aggregate

| Task | Sandbox (v1.26 baseline) | rag-system-target (v1.29) | Δ |
|---|---|---|---|
| Atomic L1 | 10/10 deterministic, ~2m | 0/10 + 4/10 (no commit), 6-8m | catastrophic regression |

**No L2'/L3' attempted.** Two atomic data points are decisive: the system has a hard scale barrier between 5 files and 91 files. More cross-file or multi-file tasks would just produce the same failure mode at higher cost. The pattern is clear:

1. Coder's `search` block hallucinates a not-quite-byte-correct version of the target file.
2. Strict applyEdits fails. Tolerant (v1.24) doesn't help (failure is structural, not whitespace).
3. Retry-with-real-content (v1.23) sometimes recovers, sometimes doesn't. At scale, more often doesn't — the LLM gets the same wrong answer twice in a row when given the literal file.
4. Validation Fixer at scale is also unreliable: 3 rounds, mostly `write failed` — Fixer also hallucinates search blocks.
5. v1.25.1 prevents task crashes (good), v1.28 surfaces partial state (good), but **the work doesn't get done**.

## What worked at scale

- **Indexing.** 91 files in 3.5s, 210 vectors, no errors. HNSW JSON survived fine. Phase 4 storage upgrade (Qdrant) is **not** the urgent bottleneck — at this scale the existing storage layer is adequate.
- **Per-step recovery (v1.3) + validation try/catch (v1.25.1) + commit_partial (v1.28).** Every infra-level safety net we built held: no task crashed, all failures were graceful with explicit signals (`commit_skipped`, `commit_partial`).
- **Reindex pruning (v1.25.2).** Worked correctly when paths changed.
- **Planner.** Correctly produced 1-step plans on both atomic tasks. Few-shot Example A/B guidance is intact at scale.

## What broke at scale

- **Coder's patch-based output.** Three independent failure paths (initial write, retry-with-feedback, validation-Fixer rounds) all produce `search` blocks that don't match the actual file. This is the dominant failure mode at scale.
- **Repo-map at default budget.** 60% of files truncated. Sort order is alphabetic, so test-file symbols dominate over production code. Model gets a misleading "what exists" view.
- **Scope creep follows broken repo-map.** Task #2 modified `orchestrator.test.ts` and created `vitest-setup.ts` — both completely unrelated to the requested change in `queue.ts`. The model's "neighbourhood awareness" came from the repo-map; with biased repo-map, neighbourhood is wrong.
- **Wall times.** 6-8 minutes for a single-file atomic task vs ~2 min on sandbox. Three Fixer rounds amplify the cost of every wrong edit.

## Lessons

1. **Patch-based output by an LLM doesn't scale beyond a small project.** This is the headline finding. Once the prompt is large enough that the model can't perfectly reproduce the file's exact byte sequence in a `search` block, every part of the patch pipeline (Coder, retry-Fixer, validation-Fixer) inherits the same problem. v1.23's whole architecture rests on the model writing exact `search` strings — at 91 files this assumption breaks. **The system needs a code-modification primitive that doesn't require the model to byte-quote existing code.**
2. **Tool-calling Coder is the right Phase 3 target.** A Coder that calls `read_file(path, lines)` and `replace_in_file(path, line_range, new_text)` (or similar) literally cannot have a "search not found" failure: it doesn't write a search string, it specifies coordinates. The current architecture's most fragile assumption is the one that breaks at scale, and tool-calling eliminates it. v1.30 in the ROADMAP, now urgent.
3. **Repo-map needs a serious upgrade before scale work.** Two issues: (a) budget too small (6KB at 91 files, would be hopeless at 500+); (b) sort order biased toward test files. v1.25 was designed for sandbox sizes. Suggested **v1.29.1** quick fix:
   - Default budget bump to ~16KB or compute from observed token count.
   - Sort priority: production paths first (anything not under `__tests__` or matching `*.test.ts`), then tests.
   - Optionally separate sections "Production:" and "Tests:" with separate budgets.
4. **Indexing/storage scales fine on this size.** Phase 4 (Qdrant migration) can wait — it would be solving a problem we don't yet have. The urgent bottleneck is the LLM-prompt format, not the vector index.
5. **Scope creep into adjacent files is a new failure mode at scale.** A direct consequence of repo-map bias. v1.29.1 sort fix should address this, but worth confirming with a re-run after the fix.

## Decision

**Phase 3, specifically v1.30 — tool-calling Coder — is now the right next move.** Empirically grounded by the data here: the system's core code-modification assumption (model emits exact `search` strings) breaks at 91 files. The fix isn't more rules, more examples, or richer retrieval — it's a different code-modification primitive entirely.

**Optional precursor: v1.29.1 — repo-map at scale.** Two-hour fix (bump budget, sort prod over tests). Improves quality of repo-map output before tool-calling Coder uses it. Also relevant if we ever revisit prompt-based Coder — won't hurt v1.30.

**Phase 4 (Qdrant, SQLite symbol table) is deferred** — indexing and vector store at 210 vectors / 91 files is not strained. Wait until ~1000+ files before investing here.

**No regression on Phase 2 wins.** All Phase 2 quality (deterministic L2.1 on sandbox, partial-completion signals, validation safety) still holds — those are sandbox-scale wins and not affected by what we found here.
