## v1.32-d.1 — mergeFixerChanges fix (landed mid-run)

**Code:**
- [packages/agents/src/orchestrator.ts](../../../packages/agents/src/orchestrator.ts) — exported `mergeFixerChanges(prior, fixer)` helper; replaces `currentChanges = fixerResult.files` with `currentChanges = mergeFixerChanges(currentChanges, fixerResult.files)`
- Semantics: Fixer wins for paths Fixer touched; paths Fixer didn't touch keep Coder's prior edit

**Tests (+4):**
- preserves Coder paths Fixer didn't touch (regression for L1.1 #1)
- Fixer wins on shared paths
- empty fixer → Coder unchanged (returns fresh array)
- empty Coder → Fixer pass-through

**Effect:** 3/3 → commit-rate after fix vs 0/1 before. Verified live on L1.1 ×3.

Not a functional regression — main change still committed cleanly. Кандидат на v1.32-d.2: либо downgrade `modify` к `create` if file absent, либо filter test-file modify in writer
- ⚠ **Diff variance** between runs: #3 byte-perfect minimal, #2/#4 add `export const app` + Tester test file. All correct, but operator-cleanable style noise. v1.32-a.6 prettier post-step не применим здесь (sandbox не имеет prettier configured)

## Lessons / next iteration ideas

1. **Bench framework is functional — operator no longer needs to run benches manually.** I (assistant) can drive Phase E end-to-end via API server + SSE stream + curl. Workflow: reset sandbox → start API → submit task → Monitor poll → capture diff/log/wall → write run-file. Acceptable to automate as `scripts/bench.sh` for Phase E continuation.
2. **Cold-load is a real factor** in first-run wall-time (+~70s). For multi-task bench sessions, pre-warm `coder` and `qwen3` aliases before the first measurement.
3. **mergeFixerChanges semantics need wider review** — works for the L1.1 case, but the current rule (Fixer wins for shared paths) might be wrong if Fixer's edit on shared path is incomplete. Worth bench against L4.1 (where Fixer DOES edit production code that Coder touched cosmetically) to verify the merge doesn't regress that path.
4. **Modify-non-existent ghost** is an orthogonal cleanup. The writer's `modify` on a non-existent file should optionally fall back to `create` if the agent meant to scaffold one. v1.32-d.2 candidate.
5. **L4.1 should be next** — it's the more interesting bench (navigation + bug-fix). If L4.1 commit-rate also ≥2/3 on llama-swap, Phase E is essentially closed and we can flip default `LLM_BACKEND=llamacpp` (Phase F).
