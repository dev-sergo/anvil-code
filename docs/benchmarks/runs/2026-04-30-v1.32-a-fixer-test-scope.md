# Run 2026-04-30 — v1.32-a Fixer test-scope discipline

Direct response to L4.1 (v1.31.2 bench). Fixer was observed editing
`tests/users.test.ts` to silence a failing `expect(user.createdAt).toBeTruthy()`
assertion, instead of fixing the production bug the assertion exercised. The
result was a green commit shipping a still-broken bug — the highest-severity
quality failure surfaced in v1.31.x. v1.32-a tightens `buildFixerAllowedSet`
so Fixer cannot write to test paths the Coder didn't already touch.

## What changed

### Code

[`buildFixerAllowedSet`](../../../packages/agents/src/tool-calling-fixer.ts):

- New constant `TEST_PATH_PATTERNS` matches three test-organization conventions: top-level `tests/`, co-located `__tests__/`, and `.test.ts`/`.spec.ts` filename suffixes.
- `isTestPath(p)` returns true if any pattern matches.
- `buildFixerAllowedSet` separates Coder's paths from issue-mention paths. After the union, paths that match `isTestPath()` AND are NOT in Coder's set are dropped. Test files Coder explicitly produced stay in scope (legitimate test maintenance).

[`FIXER_SYSTEM_PROMPT`](../../../packages/agents/src/tool-calling-fixer.ts):

- Added paragraph: "TEST FAILURES POINT TO PRODUCTION BUGS, NOT TEST BUGS." Tells the model that when a TestRunner failure cites `tests/foo.test.ts`, the assertion is the spec — the bug is in the production code being exercised. Explicitly mentions that the dispatcher will reject test-file writes that the Coder didn't touch.

### Tests

6 new unit tests in `tool-calling-fixer.test.ts` covering: top-level `tests/` drop, `__tests__/` drop, `.test.ts`/`.spec.ts` suffix drop, Coder-produced test stays in scope, non-test paths flow through unchanged, deeply-nested `__tests__` path with varying error format.

### What the Fixer attempted

The Fixer's tool sequence after Coder's first attempt failed:

```

read_file('src/routes/users.ts')                    ← retry from a different angle
replace_in_file('src/routes/users.ts', 23, 24,

```

The third tool call **is the correct fix** — the Fixer correctly identified that `UserService.create()` in `src/services/user-service.ts` is missing `createdAt`, and emitted a valid replace_in_file with the right content. **The dispatcher rejected the write** because `src/services/user-service.ts` was in neither Coder's output set (Coder only touched `routes/users.ts`) nor the issue-mention set (the TestRunner failure mentions `tests/users.test.ts`, not `user-service.ts`).

After the rejection, the Fixer fell back to a workaround in the route handler: spread the broken user and add `createdAt` post-hoc. This makes the route response look correct, but `users.set(user.id, user)` inside `create()` still stores the broken user — `UserService.list()` and `UserService.get()` will return users without `createdAt`. Validation correctly stayed red.

## What worked

- ✅ **Test-scope filter prevents test gaming.** The Fixer never edited `tests/users.test.ts` despite the TestRunner failure pointing there. Validation stayed red, commit was correctly skipped. The catastrophic L4.1 v1.31.2 outcome (green commit shipping a broken bug) is impossible under v1.32-a.
- ✅ **commit_skipped is the right signal.** Operator opens auto-branch, sees the partial workaround in `routes/users.ts`, recognizes "this didn't actually fix the bug" within seconds. No silent failure.
- ✅ **Fixer prompt + policy aligned.** The model didn't try to game the test even though it was the cheapest route to a green assertion. The prompt explicitly framed test failures as production-bug indicators, and the model followed.
- ✅ **The Fixer DID navigate to the right file.** This is informative — `read_file('src/services/user-service.ts')` happened, and the proposed edit was correct. The model can find the bug; it just can't write there under current scope.
- ✅ **6 new unit tests pass cleanly.** 362/362 green; no regression on existing scope tests.

## What broke

- ⚠ **L4.1 is not yet end-to-end fixable by the Fixer.** v1.32-a halves the failure (no longer ships a broken commit) but doesn't *complete* the fix. The path the Fixer correctly identified — `user-service.ts` — is rejected by policy because it's not in any explicit allowed-set source. The Fixer's read_file calls are not currently treated as "scope-acquisition" gestures.
- ⚠ **`extractAllowedPaths` over-restricts when error messages name a test but the bug is in production.** TestRunner failures cite the test file (where the assertion lives). With v1.32-a's filter, the issue-mention pool yields nothing usable for production-code fixes. The only reliable production-code path in scope is whatever Coder happened to write to.
- ⚠ **The Coder's first attempt was a cosmetic rename**, approved by Reviewer. This is the v1.31.2 navigation gap reasserting itself — Coder doesn't trace the bug from "POST /users" through `UserService.create()` to the storage write. Independent of v1.32-a's scope fix.

## Lessons / next iteration ideas

1. **v1.32-a is correct in intent, partial in effect.** Test-gaming is impossible; that's the catastrophic failure mode prevented. But the Fixer is now too constrained to complete L4.1 cleanly. Two natural next moves:
   - **v1.32-a.1 (small, ~1-2h):** when the Fixer (or Coder) calls `read_file(p)` during its loop, mark `p` as "explored" and add it to a per-loop dynamic allowed-write set. The model has to *deliberately* read a file to be allowed to write to it — this is a transparent, model-driven scope expansion. Forbidden patterns (`package.json`, lockfiles, etc.) still apply. Solves L4.1 directly: Fixer reads user-service.ts → can write to it.
   - **v1.32-d (larger, ~1 day):** RAG-driven Coder context. When task names a symbol/endpoint, run a code-graph "find definition + dependencies" query and inject results into the Coder's initial context. Solves the upstream navigation gap (Coder reads the right file from the start), making Fixer involvement unnecessary for tasks like L4.1.

2. **The `read_file → write_to_file` invariant is the simpler primitive.** v1.32-a.1 generalizes scope discipline elegantly: "scope grows with explicit reads." It's a single ~10-line change to the dispatcher. Would also let v1.32-d's code-graph injection work even if it adds a dozen relevant files — Coder reads what's relevant, gains write access on those, proceeds.

3. **L4.1 is now a clean test for the navigation primitive.** v1.32-a.1 should turn L4.1 from commit_skipped → committed-clean (Fixer reads user-service.ts, gets write access, fixes the bug). If it doesn't, that's evidence Coder needs upstream RAG help (v1.32-d).

4. **Reviewer is still under-powered** — it approved the cosmetic rename in attempt #1. Independent issue, possibly a Phase 4+ consideration (larger Reviewer model, or semantic-correctness check via re-running validation in dry-run before Reviewer approves). Not blocking v1.32 progress.

## Decision

**v1.32-a lands as is.** Test-gaming impossibility is the high-value safety property. The L4.1 incompleteness is honestly a *better* failure than v1.31.2's silent-broken-commit; the operator sees a partial workaround on the auto-branch and knows to investigate.

**Recommended next step — v1.32-a.1** (`read_file` grants write permission, ~1-2h, with unit tests):
- Track read paths in the dispatcher's per-loop state.
- `isWriteAllowed` consults both the static policy AND the per-loop reads.
- Forbidden patterns still take precedence (a read of `package.json` doesn't grant write — the forbidden check is non-negotiable).
- Unit tests: read-then-write succeeds; read of forbidden file followed by write still rejected; cross-loop reads don't leak.
- Re-run L4.1 — expect commit landed.

This single small change closes L4.1 *and* makes future tasks (v1.32-d code-graph, v1.32-c sub-agents) easier to scope, since "I read it, I can write it" is a clean and transparent rule the model already understands.
