# Run 2026-04-29 — v1.30 Tool-calling Coder

Phase 3 starts. v1.29 proved patch-based Coder can't survive 91 files: every
LLM call in the patch pipeline (Coder + retry-Fixer + 3 × validation Fixer)
hallucinated `search` blocks that didn't byte-match real files. v1.30 replaces
the JSON-with-search-blocks output format with a tool-calling loop:
`read_file` / `replace_in_file` / `create_file` / `delete_file` / `done`.
Coordinates instead of byte-quotes — the failure mode of v1.29 becomes
**physically impossible**.

### Critical Ollama compatibility fix

First sandbox run failed with "Tool-calling Coder produced no file changes". A direct probe to Ollama revealed:

```json
{
  "message": {
    "role": "assistant",
    "content": "{\"name\": \"read_file\", \"arguments\": {\"path\": \"src/main.ts\"}}\n{\"name\": \"done\", \"arguments\": {}}"
  }
}
```

**`tool_calls` field absent — Ollama emits tool calls inline as JSON in `content`.** Confirmed for `qwen2.5-coder:32b-instruct`; reportedly affects other Ollama-hosted models (gemma2, occasionally llama3.1). Without recovery, the loop sees zero `toolCalls`, nudges once, exits empty.

**Fix:** `extractInlineToolCalls(content)` — string-aware brace-matched JSON scanner that walks the content, extracts every `{name, arguments}` shape, returns them as if they came in `tool_calls`. 6 unit tests cover: structured field happy path, inline path, prose around inline calls, JSON in argument strings (string-aware brace counter), pure prose (no extraction), filtering of non-tool-shaped JSON. Without this fix v1.30 cannot work on Ollama at all.

## What worked

- **The v1.29 ceiling is broken.** The reason atomic tasks failed at scale on v1.29 — model writing a `search` string that didn't byte-match the file — is gone. v1.30 reads files via `read_file`, replaces by line range, and target-file edits land cleanly even on a 91-file project.
- **`/version` server.ts diff is byte-perfect surgical** — the kind of result patch-based Coder couldn't produce after 5 LLM tries.
- **Fallback parser is essential.** Without `extractInlineToolCalls`, qwen-coder produces zero tool calls because Ollama doesn't fill `tool_calls`. With it, the same model successfully invokes 5+ tools per task.
- **All Phase 1-2 safety nets stay valid.** v1.25.1 try/catch, v1.28 `commit_partial`, COMMIT_ONLY_IF_VALID — they protected both rag-system runs from landing broken commits despite scope creep.
- **WorkingSet abstraction is clean.** Lazy disk read, in-loop mutations visible to subsequent reads, `toFileChanges()` produces shape the existing pipeline already understands.

## What broke

- **Scope creep into unrelated files.** Model wandered:
  - `/version` task → wiped `package.json`, created random `vitest-setup.ts`.
  - `getSize` task → created `__tests__/` directory, again touched `vitest-setup.ts`.
  - The model treats the project as fair game for any "improvement" it imagines, not just the file the task names. With patch-based Coder this was bounded by the search-block requirement (model had to byte-quote what it touched). Tool-calling removes that bound, so the system prompt has to do the policing.
- **Structural placement errors.** `getSize()` ended up outside the class because the model didn't precisely understand which line was the class closing brace. Tool-calling helps with byte-fidelity but doesn't help with structural understanding — that requires read_file + careful replace_in_file usage. The current system prompt asks for read-before-replace but doesn't enforce "read the structural context, not just the lines you're replacing."
- **Reviewer is now under-discriminating.** First sandbox run had Coder produce 0 file changes; Reviewer **approved** that. The Reviewer agent doesn't check "did Coder actually do the task" — it just reviews patches it gets. With tool-calling, "no patches at all" is a real failure mode the Reviewer should flag. v1.30.2 candidate.
- **Wall times up** on rag-system-target (26m and 8.5m for atomic tasks; sandbox 2m). The loop iterates 5-15 tool calls with one LLM round-trip each; each round-trip on the 91-file context takes ~30-60s. Per-tool-round latency is the dominant cost.

## Lessons / next iteration ideas

1. **The hypothesis was right.** Patch-based Coder couldn't survive 91 files. Tool-calling does — atomic edits in target files land precisely. The remaining issues are different failure modes (scope, placement), not byte-fidelity.
2. **v1.30.1 — Scope discipline** (~3-4 hours). Top priority. Specifically:
   - Strengthen system prompt: explicit list of "files you may touch" derived from the step description (mentioned paths). Refuse to call create_file/replace_in_file on anything not in that list unless given explicit permission ("modify any related file").
   - Add a `forbidden_paths` allowlist: `package.json`, `package-lock.json`, `tsconfig.json`, anything matching a vitest/test-config glob — model must explicitly request to touch them.
   - Re-run `/version` and `getSize` after this — expect both to commit cleanly.
3. **v1.30.2 — Stronger Reviewer.** Reviewer must reject "Coder produced 0 file changes" as a failure unless the task description explicitly allowed no-op (extremely rare). Cheap prompt edit.
4. **v1.30.3 (optional) — Verify-syntax tool** for tool-calling Coder. After replace_in_file, the WorkingSet could brace-balance check the file. If unbalanced → tool returns warning so model retries with adjusted coords. Would have caught the `getSize` outside-class bug.
5. **Phase 4 storage upgrade still not urgent.** Indexing 91 files takes 3.5s; vector DB at 210 entries is fine. The bottleneck is LLM latency per tool round, not retrieval.
6. **`commit_partial` (v1.28) hasn't fired here even though there's "partial completion."** Both rag-system runs have target file edits + collateral damage in working tree. Auto-branch contains the changes; no commit happened (commit_skipped). So this isn't strictly "partial" by v1.28's definition — it's commit_skipped with extra working-tree mess. v1.30.1 scope discipline fix would eliminate the mess; not a v1.28 issue.

## Decision

**v1.30 lands in tree.** The empirical claim — tool-calling resolves the v1.29 patch ceiling — is decisively confirmed by the `/version` server.ts edit (perfect surgical diff at scale, vs v1.29's 0/10 cascade). The scope-creep failure mode is real but addressable in v1.30.1 with a prompt edit, not architecture.

**Default still off:** `TOOL_CALLING_CODER=true` is opt-in. v1.30.1 hardens scope discipline before flipping the default. Keeps current users (sandbox-scale) on the deterministic 10/10 patch-based path.
