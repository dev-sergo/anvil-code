# Run 2026-04-30 — v1.32-a.3 Fixer reliability + Coder retry symmetry

Direct response to v1.32-a.2's L4.1 finding: across four runs since v1.32-a,
the Fixer bailed (returned text without tool calls) on ~50% of attempts. The
single-shot retry with the "Or call done() if no source edits can fix" escape
hatch was too easy to take. v1.32-a.3 consolidates the Fixer prompt, moves
dynamic policy info to the user message where it sits next to the data it
qualifies, and replaces the one-shot retry with a two-retry pattern with
progressively stronger nudges. Coder gets the symmetric upgrade.

**Result: L4.1 closes end-to-end on the first run.** Bug-inject baseline →
Coder cosmetic edit → Reviewer approves → Validation fails → Fixer reads
the production module → Fixer applies byte-perfect fix → Validation passes
→ Commit lands with both files staged. **First operator-grade green commit
on a navigational bug-fix task.**

## What changed

### FIXER_SYSTEM_PROMPT consolidated (~40 → ~20 lines)

Sections collapsed:
- **Tools** — single block, structural-first, with one-line each
- **Workflow** — 4 imperative steps, including the "test failure → production module" navigation tactic
- **Scope** — single paragraph: read_file unlocks write; forbidden paths absolute; test paths blocked unless Coder produced
- **TS patterns** — short list of common error-message → fix mappings

Removed:
- Verbose "TEST FAILURES POINT TO PRODUCTION BUGS" block (folded into workflow step 2)
- Repetitive scope-discipline retreads — the rule appears once, not three times
- The "Or call done() if no source edits can fix" escape was killed in both prompt and retry nudges

### User message: scope-expansion adjacent to data

Before: `Allowed write targets (only these): X, Y, Z` — terminal sounding, no hint at the dynamic expansion.

After:
```
Initially-allowed write targets: X, Y, Z
Scope expansion: read_file on any non-forbidden path grants write access to it
for this loop. So if your fix needs to edit a file outside the initial list
(e.g. a service module behind a failing test), call read_file on it first —
then your edit will be allowed.
```

The model now reads the scope rule and the path list as a single coherent piece — the rule explains the list rather than living in the system prompt as an abstract constraint. Direct response to v1.32-a.2's hypothesis: prompt complexity was hiding the most important policy detail.

### No-tool-calls retry: 2 retries with progressively stronger nudges

Both Coder and Fixer now track `consecutiveNoToolCalls`; reset on any real tool call; bail only at 3.

Nudge #1 (after first text-only response):
> You responded with text but did not call any tool. Tools are how fixes happen. Pick the file the first issue references; call read_file on it; then edit. Do that now — no preamble.

Nudge #2 (after second text-only response):
> Still no tool call. The validation will not fix itself. The first listed issue points at a file path. Call read_file on that exact path RIGHT NOW. If you genuinely believe no source edit can fix the issues, call done() — but only as a tool call, not as text.

Coder has the symmetric pair, with `add_route / add_method / add_import` mentioned instead of issue navigation.

Both nudges:
- Remove the "Or call done() if there is nothing to do" escape language from the previous one-shot retry — no implicit permission to bail
- Use imperative voice ("Do that now", "RIGHT NOW")
- Mention the *concrete next call* (read_file on the file the first issue points at) so the model has a clear next step

### Tests (+5 new, 387 total)

## What worked

- ✅ **L4.1 commits cleanly on first run.** No retry needed. The pre-existing v1.32-a (test-scope), v1.32-a.1 (read-grants-write), v1.32-a.2 (commit aggregation), and v1.32-a.3 (Fixer reliability) all chained together in a single live execution to produce the expected outcome.
- ✅ **Wall time dropped from 44 min (v1.32-a.1) → 7 min.** No pathological loops. Coder did its 3 calls and finished; Fixer did its work in a focused sequence. Total ~17 tool calls instead of 40+.
- ✅ **Prompt consolidation didn't lose information.** All v1.32-a/a.1/a.2 disciplines (test scope, read-grants-write, etc.) are still enforced — but expressed in fewer, more imperative words. The Fixer behavior we wanted (navigate to production code, fix it) emerged on the first run rather than half the time.
- ✅ **Removing the "Or call done()" escape from retry nudges was decisive.** Earlier retries gave the model explicit permission to bail; the new nudges only offer "tool call now". The model picked read_file as instructed.
- ✅ **Bug fix is byte-perfect** in user-service.ts: `createdAt: new Date().toISOString()` added, `as User` cast removed, surrounding code preserved. An operator reviewing the auto-branch sees a clean, minimal diff that matches what they would have written.
- ✅ **+5 unit tests** specifically pin the new retry behavior. Future refactors that drop the retry counter, weaken the nudges, or re-introduce the "Or done()" escape will break them.

## What broke

- ⚠ **Coder still produces a cosmetic edit** to routes/users.ts that isn't strictly the bug fix (variable rename + reply.code(201).send). Not destructive — survives validation, doesn't shadow the real fix, gets committed alongside it. But it's noise. v1.32-a.3 doesn't try to suppress this — Coder's "I should do something" instinct on a navigation-required task is a separate problem from "Fixer must engage with tools".
- ⚠ **Single-run validation** — one L4.1 success doesn't establish a deterministic property; it shows the change is sufficient on this attempt. To call the workflow "robust", we'd want 3-5 consecutive successful runs. Worth doing in a separate cycle.
- ⚠ **Fixer's `read_file('user-service.js')` (wrong extension, 404)** is a small navigation cost — model guessed the wrong extension before getting the .ts right. Two-call probe instead of one. Not a bug, just inefficient.

## Lessons / next iteration ideas

1. **Removing escape language from retries was the highest-value change.** Bigger than the prompt consolidation (which was nice-to-have). The single-shot retry with "Or call done() if no source edits can fix" was an explicit "you may bail" signal to a model that, in the L4.1 case, *should not* bail. Removing it forced engagement.
2. **Imperative nudges with concrete next-step beat abstract guidance.** "Call read_file on the file the first issue references" with the issue list visible is parsed as a clear instruction. The Fixer followed it on attempt 1 of v1.32-a.3 vs ~50% of prior runs.
3. **Prompt consolidation is a maintenance practice, not just one-time cleanup.** v1.32-a/a.1/a.2 each added rules; v1.32-a.3 reorganized them. Future iterations adding rules should bias toward consolidation passes every 2-3 sub-iterations.
4. **The five-iteration arc evidence chain is itself a deliverable.** v1.31.2 (problem identified) → v1.32-a (test scope) → v1.32-a.1 (read-grants-write) → v1.32-a.2 (commit aggregation) → v1.32-a.3 (Fixer reliability) — each step measured, each gap closed, run-files documenting the chain. This is what "qualified, well-tested system" looks like in practice.
5. **Multi-run robustness benchmark** is the natural next exercise. Run L4.1 ×5 on the same baseline; expect 5 successful commits. If 4/5, document the failure variance. If <4/5, surface what broke.
6. **Coder's cosmetic-route-edit on navigation-required tasks** is the next visible imperfection. Not blocking. Possible v1.32-a.4 nudge — Coder system prompt should ask "is the task naming a file that contains the bug, or the file that *exposes* the bug?" — for navigational tasks, recognize and bail to validation+Fixer rather than producing a cosmetic edit.
