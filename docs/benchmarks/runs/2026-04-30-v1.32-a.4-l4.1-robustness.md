## Findings

### What's now robust

- **End-to-end pipeline reliability.** Bug-inject → Coder → Reviewer → Validation → Fixer (when needed) → Validation pass → commit landed → working tree clean. This sequence ran 5 times without a single bail or rollback.
- **Fixer engagement.** v1.32-a.2 saw 0 tool calls in 2 consecutive Fixer attempts (~50% bail rate); v1.32-a.4 has 0/5 bails. The no-tool-calls retry + prompt consolidation from v1.32-a.3 hold up across runs.
- **Bug fix correctness.** All 5 runs add `createdAt: new Date().toISOString()` to the user object literal AND remove the `as User` cast. Every test asserting `expect(user.createdAt).toBeTruthy()` passes.
- **Commit aggregation (v1.32-a.2).** Every commit hash is real (not empty). Working tree clean after each run. Multi-file commits stage both Coder's path and Fixer's path correctly.

### Visible quality variance

- **Diff quality is non-deterministic.** Only 1/5 runs produces a byte-perfect minimal diff (run #4). The other 4 runs introduce some cosmetic noise. This is *not* a regression — it's the same model variance that exists across any LLM-driven task. The system pipeline is deterministic; the model's specific edits are not.
- **Wall-time variance is large.** 5 min (clean run) vs 58 min (Coder loop pathology) is a 12× spread. p90 ≈ 17 min, p99 unknown without more samples. The 58-min outlier is the most concerning; it's a real wall-time tax the operator pays.

### What's still imperfect (independent of v1.32-a.4)

- **Coder loop pathology** (run #1, 58 min). When Coder picks a replace_in_file edit that fails brace-balance, it sometimes spins retrying near-identical alternatives. Not destructive (eventually exits, hands off to Fixer who fixes the actual bug), but expensive.
- **Coder routinely produces a cosmetic edit on routes/users.ts** even when the real fix is in services/user-service.ts. Doesn't shadow the Fixer's fix; just adds noise to the diff. Same as v1.32-a.3's observation.
- **Style consistency depends on the model's mood**. 4 of 5 runs introduce minor style noise (indent shifts, missing trailing commas, blank lines). A `prettier --write` post-step in the orchestrator would normalize this — possible v1.32-a.5.

## What worked

- ✅ **Robust commit-landed rate.** 5/5 runs produced a green commit on the auto-branch with the bug semantically fixed. The pipeline does what it claims.
- ✅ **No bails, no rollbacks, no destructive edits.** 0/5 cases of Fixer producing 0 tool calls. 0/5 cases of validation forcing commit_skipped after Fixer attempted a fix. The retry+prompt fixes from v1.32-a.3 hold up across all 5 runs.
- ✅ **Model variance moves to the *quality* layer, not the *correctness* layer.** Every run was correct (compiled, tests passed, bug fixed). Variance is in *how clean the diff is*, not *whether the fix is right*. This is the desired failure profile — operator gets a working fix to review, with at most some style noise to clean up.
- ✅ **Median wall-time is 6 min.** Fast on the typical case. Acceptable for an autonomous bug-fix loop on a real-size sandbox project.

## What broke

- ⚠ **Run #1 took 58 minutes** because the Coder phase entered a brace-balance retry loop. Same pattern v1.32-a.1 surfaced. Independent of the v1.32-a.x arc, but visible in this robustness sample. Worth attacking in a future iteration.
- ⚠ **4/5 runs introduce cosmetic noise** in the diff. Not breaking anything, but the diff isn't always merge-ready as-is. A `prettier`/`eslint --fix` post-step in the orchestrator would solve this for free; tracked as a possible v1.32-a.5 micro-improvement.
- ⚠ **Coder still produces a cosmetic edit on routes/users.ts** in most runs even when the real fix is elsewhere. Adds noise to the multi-file commit. Discussed in v1.32-a.3 — not a blocker.

## Lessons / next iteration ideas

1. **Robust ≠ deterministic.** The pipeline is now robust at the workflow level (5/5 commits land), but the LLM is still non-deterministic at the diff level (1/5 byte-perfect). This is the natural separation: engineer the *pipeline* for reliability, accept that the *model output* will vary. Operator review remains in the loop.
2. **The v1.31.2 → v1.32-a.4 arc is complete.** L4.1 was the canary task for navigational bug-fix workflows; it now lands consistently. The architectural foundation (tool-calling Coder, structural anchors, scope discipline, read-grants-write, commit aggregation, Fixer reliability) all chain together as designed.
3. **Coder loop pathology is the next visible bottleneck** for wall-time. Run #1's 58 min is unacceptable as a worst case for an autonomous loop. **v1.32-a.5 candidate (~2-3h):** detect repeated identical-or-near-identical tool calls in the Coder loop; after N (say 5) repetitions break early with a system reminder ("you've retried this edit N times — change strategy or call done"). Should pull the long-tail wall-time down significantly.
4. **`prettier --write` in orchestrator post-step** is essentially free quality improvement — a sandbox project with prettier configured would have its formatting normalized after every Coder/Fixer write. Convert run #2/#3/#5's "minor style" issues into byte-perfect outcomes for free. **v1.32-a.6 candidate (~1h)**.
5. **Phase 3 closure is now realistic.** With L4.1 stable, the natural next big step is **v1.32-c — sub-agents** (BugFixAgent / RefactorAgent / FeatureAgent), each with a smaller specialized toolset. The L4.1 evidence shows the underlying primitives work; sub-agents would let the planner pick the right specialist instead of one Coder + one Fixer doing everything.
6. **Phase 4 is now within reach.** With Phase 3 mostly complete, the storage upgrades (Qdrant, SQLite symbol table) and backend swap (llama.cpp direct with grammar-constrained tool calls) become the meaningful productivity levers.

## Decision

**v1.32-a.4 lands** — no code changes, just bench evidence. The L4.1 arc is decisively closed at the workflow level. The remaining variance is at the LLM-output-quality layer, which is the right place for it.

**Recommended next step — chosen from priority order:**

- **v1.32-a.5** (Coder loop-pathology guard, ~2-3h) — pull the long-tail wall-time down. High value if the operator actually wants to use the system as a daily driver.
- **v1.32-c** (sub-agents, ~1 week) — Phase 3 architectural closure. The bigger investment with more upside.
- **v1.32-a.6** (`prettier` post-step, ~1h) — cosmetic cleanup, very cheap.

Reasonable order: **a.5 first** (catches a real bottleneck that affects all benches), **a.6 in parallel or skip** (small enough to do whenever), **then v1.32-c**. After Phase 3, the natural next direction is Phase 4 (storage + backend).
