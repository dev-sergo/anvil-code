## What changed

### Code

[`WorkingSet.hasOpened(relPath)`](../../../packages/agents/src/working-set.ts) — public method returning `true` if the file is in the working-set cache (read or modified) and not deleted. The existing lazy-load `read()` already populates the cache as a side effect, so this is a free signal — no new tracking machinery, just a getter on existing state.

[`isWriteAllowed(path, policy, ws?)`](../../../packages/agents/src/tool-calling-coder.ts) — added optional `ws` parameter. Logic:
1. Forbidden patterns take precedence (absolute ban — reading `package.json` does NOT grant write).
2. Static allowlist hit → allow.
3. `ws.hasOpened(path)` → allow (the v1.32-a.1 dynamic unlock).
4. Otherwise reject with a message instructing the model to `read_file(path)` first.

All four dispatcher write-paths (`replace_in_file`, `create_file`, `delete_file`, `executeStructuralEdit`) now pass `ws` through. The `ws` parameter is optional for backwards-compat; existing isolated unit tests without WorkingSet instances still work.

[`FIXER_TEST_PATH_FORBIDDEN`](../../../packages/agents/src/tool-calling-fixer.ts) — Fixer's policy now combines config-file forbidden patterns (package.json etc.) with test-path patterns. Closes the read-grants-write loophole that would otherwise re-open v1.31.2's L4.1 game-the-test path: read `tests/foo.test.ts` → gain write access → mutate the assertion. With test paths in the forbidden list, the read-unlock is gated by `!policy.allowed.has(path)` — only Coder-produced tests stay writable; ad-hoc reads of unrelated tests don't grant write.

[Coder + Fixer SYSTEM_PROMPTs](../../../packages/agents/src/) — explain the read-grants-write rule to the model: rejected writes mention "use read_file first" so the model knows the resolution path.

### Tests (+18 new)

### Two adjacent issues surfaced

#### 1. Coder phase pathology (44-minute loop)

In the Coder phase, the model called `replace_in_file('src/routes/users.ts', start_line=18, end_line=23, ...)` ~30+ times in a row, alternating between near-identical variants. Tracing:

- File line 17 starts `app.post('/users', async (request, reply) => {` (an opening brace).
- Coder's replace targets lines 18-23 (the body inside the existing `app.post` block).
- But the model's `new_text` STARTS WITH `    app.post('/users', async (request, reply) => {` — duplicating the wrapper that's already on line 17.
- This produces an unbalanced result (one extra opening `{` from the duplicate header). v1.30.5's brace-balance check rejects → WorkingSet rolls back → model retries with the same shape → eternal loop.

The model never figured out that line 17 already had the `app.post` declaration. It kept including the wrapper in `new_text`.

**This is independent of v1.32-a.1** — it's a Coder navigation+intent issue analogous to v1.31.2's L4.1 finding (Coder doesn't trace through the file structure correctly). v1.32-a.1 doesn't claim to fix Coder navigation; it fixes Fixer scope.

The Coder eventually called `done()` close to MAX_TOOL_CALLS (50). All its rolled-back edits left routes/users.ts at baseline on disk. SafeWriter wrote routes/users.ts as a no-op (or near-no-op — log says `changedLines: 2` but `git diff` shows no change vs main).

**Wall-time impact** is real — 44 min on a 5-file sandbox task is unacceptable. Worth investigating in a follow-up. Possible mitigations:
- Detect "Coder is calling the same tool with the same args repeatedly" and break the loop early
- Tighter MAX_TOOL_CALLS budget (50 → 25 for Coder, mirroring Fixer's v1.30.3.1 reduction)
- A cooler-headed system-prompt nudge after N rejected edits ("you've been rejected N times — try a different approach")

#### 2. Commit didn't land despite "Validation passed"

The orchestrator logged `Committed changes` with `commitHash: ""` (empty), and `Task completed`. But `git log` on the auto-branch shows HEAD unchanged from the bug-inject commit, and `git status` shows `user-service.ts` modified-but-not-staged.

The Fixer's correct fix is in the working tree but NOT committed. An operator looking at the auto-branch would see no progress (HEAD identical to base), and only by running `git status` would notice the dirty tree.

This appears to be a git-engine or orchestrator bug — possibly:
- `commitChanges` was called with `files: []` (Fixer's FileChange[] not propagated correctly)
- simple-git's `add([])` is a no-op, then `commit` against an unstaged tree returns no commit
- Empty hash → orchestrator logs "Committed changes" with the empty value as if successful

Worth investigating: which path in `runValidationLoop` aggregates Fixer's FileChanges into the commit's file list? If it only lists Coder's files (which were no-ops here), the commit ends up empty.

**Independent of v1.32-a.1** — this is a separate bug that would have happened regardless of the read-grants-write change. Notable as a distinct finding for next iteration.

## What worked

- ✅ **`read_file → write` chain works.** The Fixer read user-service.ts, gained write access, applied a correct fix. Without v1.32-a.1, the same exact write was rejected (v1.32-a). The architectural change is empirically validated.
- ✅ **Forbidden patterns still absolute.** Tests verify reading `package.json` doesn't grant write. The new dynamic scope expansion respects the static absolute ban.
- ✅ **Test-path forbidden closes the loophole.** Fixer can't game tests via the new read-grants-write path either. v1.32-a's test-scope discipline holds.
- ✅ **Structural-tool integration.** `add_method` and other structural tools also benefit from read-grants-write (verified by integration test). The locator runs against the WorkingSet; once read, structural edits flow through normally.
- ✅ **+18 unit tests** — six dimensions of coverage on a single architectural primitive: WorkingSet state, isWriteAllowed logic, dispatcher integration, Fixer policy, isolation, backward-compat. Tight coverage for a small change.
- ✅ **Fixer applied the right fix on the bug-inject baseline.** Working tree now has `createdAt` set in `UserService.create()` and the `as User` cast removed. End-state code is correct.

## What broke

- ⚠ **Coder phase pathological loop (44 min).** Independent of v1.32-a.1, but newly visible because the bench took so long. Coder kept retrying near-identical `replace_in_file` calls that brace-balance kept rejecting. Worth a v1.32-a.2 or v1.32-c follow-up: detect repeated identical tool calls and break the loop with a system reminder.
- ⚠ **Commit didn't land despite "Validation passed".** Working tree has the correct fix; auto-branch HEAD is unchanged. Looks like the orchestrator's commit step isn't getting Fixer's FileChange[] into the staged set. Independent bug.

## Lessons / next iteration ideas

1. **Read-grants-write is the right shape of the dynamic scope rule.** Transparent, model-driven, low-ceremony. The model gains write permission only by deliberately reading — there's no way to "sneak" writes to a file without first declaring intent via read_file. Forbidden patterns still gate the most dangerous paths absolutely. This is the level of permissioning that matches how a careful human developer works.
2. **The next iteration's natural focus is the orchestrator/commit-landing issue.** v1.32-a.1 produced a correct fix that didn't ship. **v1.32-a.2 (~1-2h)**: trace the FileChange[] flow through `runValidationLoop`, ensure Fixer's changes are in the commit's staged set, add a test that "Fixer fix → commit lands" on a sandbox bug-fix scenario.
3. **Coder loop pathology on L4.1** is visible because the task is unusually demanding (model needs to navigate through routes/users.ts to user-service.ts). The 44-minute wall time is unacceptable. **v1.32-a.3 (~2h)**: detect repeated-identical tool calls in the Coder loop, break early with a "you've been rejected N times — change strategy" prompt nudge. Or tighten MAX_TOOL_CALLS for Coder.
4. **`replace_method_on_object_literal` is a legitimate gap** — but small. Most v1 codebases have classes, not const-object-literal services. If the user's project leans on object-literals (or factory functions), this becomes more relevant. Defer to v1.33+.
5. **The four-iteration arc v1.31.2 → v1.32-a → v1.32-a.1 → (next) is a clean evidence chain** — each step measured against the same L4.1 bench, with progressively cleaner outcomes. v1.31.2 silent broken commit → v1.32-a visible incomplete fix → v1.32-a.1 correct fix in working tree. A v1.32-a.2 commit-landing fix would close it: correct fix in committed auto-branch.
