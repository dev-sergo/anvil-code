## What worked

- **Pruning unblocks short Fixer sessions.** First time a tool-calling Fixer attempt has completed on this 91-file project without Ollama dying. The architectural fix is sound — when a Fixer attempt converges in 1-2 rounds and stays under the threshold, pruning isn't even needed. When it goes deeper, pruning kicks in and keeps the prompt manageable.
- **Smaller MAX_TOOL_CALLS (25)** is consistent with the observation that Fixer attempts shouldn't take 50 rounds. If 25 isn't enough, the Fixer probably can't fix this issue from validation output alone — better to commit_skipped than burn tokens.
- 280/280 unit-tests green; pruning logic covered.

## What broke

- **Fixer can't converge on cargo-culted /version.** The Coder writes a /version handler that's a copy of /health — wrong shape. Fixer reads the file, but typecheck/test errors don't pinpoint "the return value is wrong" — they say "type assertion failed" or whatever. Fixer doesn't know what the correct return value should be without re-reading the original task description, which it doesn't have direct access to. So Fixer iterates without converging, and validation attempt #2 is fundamentally a "different file edit, same problem" loop.
  - **This is a pipeline composition problem, not v1.30.3.1.** v1.30.4 (Coder cargo-culting fix) addresses the upstream cause: Coder shouldn't produce wrong content in the first place.
- **Ollama crashes on long single-attempt sessions still observed.** Attempt #2 ran 8 minutes alone (not cumulative — fresh WorkingSet). So pruning is necessary but not sufficient. The llama runner can crash mid-session on a long-enough load even when the conversation is small.
  - Possible mitigations: `fast` mode (smaller model), shorter context windows in Ollama config, or simply accepting this as a known infra limit when Fixer can't converge quickly.

## Lessons / next iteration ideas

1. **Pruning is the right primitive but not enough alone.** When Fixer can converge in <5 rounds (small total work), it works. When the underlying issue is unfixable from validation output (cargo-cult content), Fixer iterates pointlessly until Ollama crashes.
2. **Cargo-culting fix (v1.30.4) is now the higher-priority unblock.** If Coder produces correct content, Fixer doesn't need to run at all — validation passes on attempt 1. Cheap prompt change addressing the root cause.
3. **Reviewer rejects empty Coder output (v1.30.2)** still pending; cheap, useful, independent.
4. **Optional: skip Fixer when validation issues don't reference the changed files.** If typecheck output is from project-wide tooling (e.g. tsc errors elsewhere in the repo not introduced by Coder's change), Fixer can't fix them anyway. Detection: `issues_files_set ∩ coder_changed_files == ∅` → skip Fixer, go straight to commit_skipped.
5. **Smaller LARGE model for Fixer?** qwen2.5-coder:14b might handle these long sessions better than 32b. Tradeoff: less capable per-round, but Fixer is a constrained task ("address listed issue with minimal edit") where a smaller model could suffice. v1.31+ candidate.

## Decision

**v1.30.3.1 lands in tree.** Architectural fix is correct, unit-tested, and produced a measurable improvement: first-ever completion of a tool-calling Fixer attempt on a 91-file project. Full bench convergence is blocked downstream (cargo-culting → Fixer can't recover) — the next concrete unblock is v1.30.4 Coder prompt fix, not more Fixer changes.

**Sequence:**
- **v1.30.4** (~30 min) — Coder prompt: don't cargo-cult sibling routes, derive content from task description
- **v1.30.2** (~1h) — Reviewer rejects empty Coder output
- After both: re-run /version + getSize. If Coder produces correct content, Fixer should rarely fire, the Ollama-crash failure mode goes away, and we should see the first end-to-end GREEN commit on rag-system-target.
