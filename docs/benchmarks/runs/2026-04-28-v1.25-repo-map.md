## What v1.25 changed (under test)

- New module [packages/code-graph/src/repo-map.ts](../../../packages/code-graph/src/repo-map.ts):
  - `buildRepoMap(graph, projectRoot, opts?)` — pure render of the in-memory CodeGraph snapshot
  - Per-file: relative path → list of symbols → for class/variable, indented method signatures pulled from `text` via regex (no AST re-walk); for interface, indented field lines
  - Token budget (default 6000 chars ≈ 1500 tokens) with greedy fill
  - `highlightFiles` — pinned at top, never truncated; entry points + paths from `previousChanges` (cross-step state) feed this
  - Control-flow keywords (`if/for/while/...`) filtered out of method extraction
- `PromptContextInput.repoMap?: string` — new optional field rendered as the *second* prompt section, between Project Conventions and Recently-modified/Existing-files
- New `GraphRetriever.graph` getter — exposes the live CodeGraph for renderers without a separate snapshot
- `Orchestrator.renderRepoMap(extraHighlights)` helper — built fresh on every call (cheap), used by Planner / Architect / Coder / Reviewer / retry-Fixer / validation-Fixer prompts
- 10 new unit tests in `repo-map.test.ts`; 215/215 across the suite; 12/12 packages build

### L2.3 — Run #2

Initial submission failed instantly with `fetch failed` after a 4-hour idle window — Ollama transient connectivity issue, not v1.25-related. Reissued; the retry started cleanly and ran for ~7m 30s before crashing the *whole task* (status `failed`, not `commit_skipped`):

```
Job failed: SafeWriter.execute: edit-apply failed for src/routes/users.ts:
edit #1: search string not found in file: const body = request.body as ...
```

**What happened:** initial Coder edits for all 3 files applied cleanly, tests passed, **typecheck failed**. Validation Fixer (attempt 1) generated edits whose `search` blocks didn't match the post-Coder file contents. `applyEdits` errored, `SafeWriter.execute` threw, the throw bubbled up through `runValidationLoop` → `runTask` → JobWorker as an uncaught exception.

**This is a pre-existing orchestrator bug, not a v1.25 regression.** [packages/agents/src/orchestrator.ts:622](../../../packages/agents/src/orchestrator.ts#L622) calls `this.writer.execute(fixed)` inside the validation loop with no try/catch — only the *initial* write phase is guarded. v1.24 didn't surface it because Fixer's edits then happened to apply (with a poor result, but still applied). With repo-map producing different Coder output, the validation Fixer's quoted text drifted enough to miss strict-match.

Fix is small (wrap `writer.execute(fixed)` in try/catch and treat throws like another validation issue rather than crashing the task). Tracking as **v1.25.1 follow-up** in ROADMAP.

**Score:** N/A — task failure was infrastructure-level, not output-quality.

## What worked

- **L2.3 cumulative produced its first GREEN commit ever** (9.2/10), with clean tri-file coherence and even a senior-engineer touch (extracting `UserService.delete()` as a method). This is the single most important data point in the run — repo-map directly enabled it. Prior best on this task was 5.0/10 with silent partial commit.
- **L2.1 clean now lands a commit in both runs** (was `commit_skipped` on the v1.24 baseline). The mean score jumped from 5.2 (single-shot) to 6.4 (mean of N=2).
- **Orchestrator integration is clean.** The repo-map is built on demand from a public `GraphRetriever.graph` getter; the per-prompt overhead is ~450 bytes for the sandbox (≈100 tokens) — well below the 1500-token budget.
- **No regressions on L1.1.** Two identical-diff runs proves repo-map doesn't perturb simple atomic tasks.

## What broke

- **High variance on L2.3 cumulative.** 1 of 3 runs landed cleanly; one hit an orchestrator bug; one died from cross-step Planner inconsistency. Even the best run depended on Planner choosing a 1-step plan (rule 6a). When Planner emits 2 steps, the same prompt rules don't pull the model through.
- **L2.1 still picks the wrong Fastify hook** — both runs used `onRequest` despite `onResponse` being verbatim in the task. Repo-map can't help here because Fastify's hook signatures aren't in the sandbox's indexed source. This is a v1.26 (few-shot examples) target, not a v1.25 one.
- **Validation-Fixer write throws crash the whole task.** [orchestrator.ts:622](../../../packages/agents/src/orchestrator.ts#L622) — pre-existing, surfaced for the first time on L2.3 #2. Needs a try/catch and same partial-success treatment as the initial-write phase. Tracking as **v1.25.1**.
- **Stale graph on `git reset --hard`.** Discovered while preparing the runs: a manual `/index` reindex is delta-only (uses cached file-hashes, never removes deleted files from the graph). After `git reset` removed `request-log.ts`, the graph still contained its `requestLogMiddleware` symbol → repo-map advertised a "ghost" file the model treated as already existing → L2.1 silently skipped creating it. Worked around by deleting `data/projects/<id>/` and re-indexing from scratch. Tracking as **v1.25.2**: full reindex should diff against discovered files and prune stale graph entries.

## Lessons / next iteration ideas

1. **Repo-map breaks the cumulative ceiling — when it gets a chance.** The 9.2/10 result on L2.3 cumulative is unambiguous: with the live symbol inventory in the prompt, the model stops hallucinating and lands coherent multi-file changes. This is the v1.0 target on cumulative tasks.
2. **Planner is now the main variance source on cumulative tasks.** When it emits a single coupled step, the model is fine; when it splits, cross-step drift wins. v1.27 (per-agent context tailoring) should give Planner a stronger steer toward 1-step cross-file plans.
3. **Two orchestrator hardening items uncovered, both small:**
   - **v1.25.1**: try/catch around `writer.execute(fixed)` inside `runValidationLoop`. On apply-fail, surface as an issue and continue rather than crashing the task.
   - **v1.25.2**: full `/index` reindex should compute set-difference against on-disk files and prune graph entries for vanished paths. Otherwise repo-map can confidently lie about ghost files post-`git reset` / file deletion.
4. **Few-shot examples for Coder (v1.26) is the right next iteration.** L2.1's hook-misuse pattern across both runs is the textbook case where one worked example beats four prose rules.
5. **Tolerant fallback (v1.24) still hasn't fired in vivo.** Two iterations of benchmarks, zero activations. Consider it permanent insurance — leave in tree, don't expect it to swing scores.

## Decision

Land v1.25 in tree. The L2.3 cumulative result alone (9.2/10 GREEN, prior best 5.0/10) clears the bar set in the project ROADMAP for this iteration. Variance and the two orchestrator/indexing follow-ups are real but small (each <1 day).

Next: ship **v1.25.1** (validation-Fixer try/catch) + **v1.25.2** (reindex prunes deleted files) as quick hardening, then move to **v1.26** (few-shot examples) — the data here points clearly at it.
